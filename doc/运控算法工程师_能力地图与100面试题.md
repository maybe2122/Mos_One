# 📍 Part 1 · 能力地图

> 成为运控工程师要会什么 · 本仓库哪里实现 · 还缺什么

### 1️⃣ 数学与刚体建模基础 🟢

- **要会**：旋转表示（旋转矩阵/四元数/欧拉角及奇异性）、坐标系变换、刚体速度/角速度、雅可比、李群 SO(3)/SE(3) 入门。
- **本仓库**：obs 用 `projected_gravity_b`（重力在 base 系投影）表姿态而非欧拉角 —— `env._get_observations`；速度全部取 base 系（`root_lin_vel_b`/`root_ang_vel_b`）。
- **缺口**：无显式 floating-base 动力学（M、C、G 矩阵）建模，靠仿真器代劳。

### 2️⃣ 腿足运动学 FK/IK 🟢🟡

- **要会**：串联腿 FK/IK 解析解、可达空间、多解分支、雅可比与 Jᵀ 力映射。
- **本仓库**：`deploy/common/kinematics.py` —— 3-DOF 串联腿（髋外摆 q_ab + 大腿 + 膝），解析 IK 含 `knee_sign` 分支与可达性 clamp；`--selftest` 往返误差 1e-15。L_thigh=0.180m。
- **闭链特例**：膝是平行四连杆，模块输出"约定角"，下发前需仿射映射 `q_motor = sign·q + offset` 标定。

### 3️⃣ 步态规划 🟡

- **要会**：支撑/摆动相、相位与占空比、足端笛卡尔轨迹、对角 trot/bound/pace/pronk。
- **本仓库**：`deploy/common/gait.py` —— 对角小跑（FL+RR 同相、FR+RL 反相），摆动相**摆线轨迹**（触地/离地瞬间水平+竖直速度均为 0，少打滑），笛卡尔规划→IK→关节目标。
- **关联**：RL 侧不显式给步态，而用 `gait_symmetry`/`anti_bound` 奖励"涌现"出 trot。

### 4️⃣ 机器人动力学与执行器选型 🟡🔴

- **要会**：Newton-Euler/Lagrange、关节力矩需求估计、电机 T-N 曲线、减速比/反驱/热、CoT。
- **本仓库**：`deploy/common/dynamics.py` + `doc/dynamics_gear_ratio_analysis.md` —— 矢状面静力递推得**峰值≈12 N·m**；减速比可行带 [3.56, 10.66]，**最优 N\*=6.16**，现用 6.33 已接近最优。电机 GO-M8010-6（23.7 N·m / 30 rad/s @24V）。
- **核心结论**：真机"力矩不足"根因不是减速比，而是 `effort_limit_sim=12` 零余量 + 驱动器电流上限/母线掉压。

### 5️⃣ 强化学习理论 🟡🔴

- **要会**：MDP、on/off-policy、策略梯度、PPO（clip 目标）、GAE、actor-critic、探索-利用。
- **本仓库**：`agents/rsl_rl_ppo_cfg.py` —— ActorCritic `[256,256,128]` ELU；PPO clip 0.2、entropy 0.01、γ0.99、λ0.95、adaptive LR(desired_kl 0.01)、24 steps×4096 env、5 epochs×4 minibatch。

### 6️⃣ RL 环境 / MDP 设计 🟡🔴

- **要会**：观测空间（含/不含特权信息）、动作空间与控制模式、终止条件、reset 与 episode 设计。
- **本仓库**：`mos2026_2_closed_usd_env.py`（DirectRLEnv）—— **45维 obs** = root_lin_vel(3)+ang_vel(3)+proj_gravity(3)+(jpos−default)(12)+jvel(12)+last_action(12)；**12维动作**（残差式 `default+scale·clip(a,±1.5)`），position/effort 双模式；decimation=4（物理200Hz/策略50Hz）。
- **缺口/坑**：cfg 注释把 command 写进 obs 但代码没有 → 见 Q17（固定前进指令、不可在线改速）。

### 7️⃣ 奖励塑形与步态病理 🔴（本仓库强项）

- **要会**：稀疏 vs 稠密、shaping 导致的局部最优、步态病理（原地不动 / 跳着走 / 滑步）与对症奖励。
- **本仓库**：`reward_scales`(env_cfg) + `_compute_*`(env) + `doc/足端打滑问题.md` —— track_lin_vel_xy(exp,σ=1.0,w6.0)、flat_orientation(−2.5)、base_height(−3,clamp)、foot_slip(−0.1)、**对角 gait_symmetry(−0.5)**、**anti_bound(−1.0)**、torque(opt-in)…；去掉 alive 破"原地不动"，对角相位+anti_bound 破"跳着走"。

### 8️⃣ 仿真器与闭链工程 🔴

- **要会**：PhysX vs MuJoCo 接触/约束求解、并行化、GPU 显存、闭链/并联机构。
- **本仓库**：4 个 `*_close_loop` 关节 `excludeFromArticulation`，需 `enableProjection` 否则 anchor 漂移→**1e28 reward 尖峰**（`projected_loop_joint_names`、`_patch_projected_loop_joints`）；`replicate_physics=True` 快克隆要求 USD 无嵌 PhysicsScene（`tools/asset/strip_embedded_ground.py`）；PhysX GPU caps 按显存收紧。

### 9️⃣ 域随机化与 Sim2Real 🔴

- **本仓库**：`EventCfg` —— startup(摩擦0.6–1.4 / base±质量 / 腿±20%)、reset(Kp/Kd±20%、零位±0.05rad)、interval(推搡±0.5m/s)、obs 高斯噪声；`--domain_rand` 开、eval 关（干净测量）。

### 🔟 状态估计 / 特权信息 / HIM 🔴

**Q1 🟢** 旋转的三种表示（旋转矩阵 / 四元数 / 欧拉角）各自优缺点与奇异性？本仓库 obs 为何用 `projected_gravity_b` 表示姿态，而不直接给四元数或欧拉角？
　🎯 考点：万向锁、四元数双覆盖、姿态信息的最小充分表示、对 yaw 不变性。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

旋转矩阵：9 参数无奇异、复合方便，但冗余（6 个正交约束）、不利于做网络输入/插值。四元数：4 参数无万向锁，但有**双覆盖**（q 与 −q 同一旋转），直接喂网络会在 ±q 处不连续。欧拉角：直观、最少参数，但**万向锁奇异**（pitch=±90°）且顺序约定易错。

`projected_gravity_b = Rᵀ·[0,0,−1]`：① 连续光滑、无双覆盖问题；② 只含 roll/pitch 信息、**对 yaw 天然不变**——平地行走任务本就该 yaw 等变，少给一维冗余信息能提升泛化；③ 物理上就是"加速度计静态读数"，与真机 IMU 可得量对齐。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q2 🟢** 推导 3-DOF 串联腿（髋外摆 + 大腿 + 膝）的解析逆运动学。为什么会有两个解分支（`knee_sign`）？工程上如何选分支并处理不可达目标？
　🎯 考点：2 连杆余弦定理、肘上/肘下、可达性 clamp。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

髋外摆 q_ab 先解耦：足端目标绕髋滚转轴旋回矢状面，q_ab 由 atan2 分量直接得出。剩下平面 2 连杆：余弦定理 `cosθ_knee = (r² − L1² − L2²)/(2L1L2)`，acos 有 ± 两支 → **肘上/肘下**两种构型，即 `knee_sign`。

工程选支：按机械限位和默认站姿选定一支并**全程锁定**（避免轨迹中途跳支导致关节速度突变）。不可达处理：r 超出 [|L1−L2|, L1+L2] 时把目标 clamp 回可达球壳（仓库里 clamp 到边界再解），保证任何输入都返回有限解。`--selftest` 用 FK∘IK 往返误差 1e-15 验证解析解正确性。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q3 🟡** 什么是腿部雅可比？如何用 `τ = Jᵀ·F` 把期望足端力映射成关节力矩？奇异位形（腿伸直）会带来什么数值问题？
　🎯 考点：微分运动学、J 的条件数、力控基础。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

雅可比 J 把关节速度映成足端速度 `v = J q̇`；由虚功原理 `τ = Jᵀ F`，期望足端力（如支撑反力）直接映成关节力矩，是力控/柔顺控制的基础。

腿伸直时 J **降秩**（奇异）：条件数爆炸，逆运动学速度解 `q̇ = J⁻¹v` 数值发散；力的某些方向需要无穷大力矩（沿腿轴方向的力此时不需要膝力矩，但垂直方向失去操纵能力）。对策：阻尼最小二乘（DLS）、轨迹规划上避免完全伸直（留弯膝裕度）。

</div>
</details>

**Q4 🔴** 本机器人膝是平行四连杆闭链，`kinematics.py` 输出"约定角"，但电机实际驱动的是 `*_shank_link`。解释"约定膝角 ↔ 电机轴角"为何不是恒等映射，如何标定 `q_motor=sign·q+offset` 这条传动？
　🎯 考点：闭链传动比、零位标定、sim/real 关节映射。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

平行四连杆把膝部电机（装在大腿根/髋部附近）的转动经四杆传到 `*_shank_link`。"约定膝角"是几何上定义的大腿-小腿夹角，电机轴角是编码器读数：两者差在 ① 零位定义不同（装配零位 vs 几何零位 → offset）；② 旋向约定可能相反（→ sign）。平行四连杆传动比恒为 1，所以是仿射而非非线性映射。

标定：把腿摆到若干已知几何姿态（如用工装/水平仪定大腿小腿角），记录电机编码器值，最小二乘拟合 sign 与 offset；再用 FK 预测足端位置 vs 实测交叉验证。注意 GO-M8010-6 编码器掉电只丢整圈（2π 模数），圈内 offset 一次标定长期有效。

</div>
</details>

**Q5 🟡** Newton-Euler 与 Lagrange 建模的区别？`dynamics.py` 用矢状面静力递推估出峰值 ~12 N·m，动态蹬地相比静立多出的力矩主要来自哪些项？
　🎯 考点：递推 vs 能量法、惯性力/科氏力/重力项。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

Newton-Euler：逐连杆递推力/力矩平衡，O(n)、数值化、适合在线控制与逆动力学；Lagrange：能量法，给出 `M(q)q̈ + C(q,q̇)q̇ + G(q) = τ` 的解析结构，适合分析与控制律设计。两者等价。

`dynamics.py` 静力递推只含 **G 项**（重力静载）。动态蹬地多出的力矩来自：① 惯性项 M q̈（加速躯干与摆腿）；② 科氏/离心项；③ **支撑腿减少时的载荷重分配**（trot 只有 2 腿着地，单腿承重翻倍）；④ 蹬地相地反力 > 体重份额（要产生向上/向前加速度）；⑤ 接触冲击瞬态。所以峰值取静力的 2–3 倍裕量来选电机。

</div>
</details>

**Q6 🔴** 给定电机 T-N 曲线与负载需求，如何确定可行减速比区间与最优值（本仓库得 [3.56,10.66]、N\*=6.16）？增大减速比对峰值力矩、最高速度、反驱性、折算惯量分别有何影响？
　🎯 考点：T-N 包络、力矩/速度权衡、反射惯量 ∝ N²。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

约束两条：力矩 `N ≥ τ_need/τ_motor`（下界 3.56）、速度 `N ≤ ω_motor/ω_joint_need`（上界 10.66）。最优点常取**利用率均衡**（力矩裕度与速度裕度相等）或按反射惯量匹配 `N* = √(J_load/J_motor)`，本仓库得 N*=6.16，现用 6.33 已贴近。

增大 N：峰值关节力矩 ∝N ↑；最高关节速度 ∝1/N ↓；**反驱性变差**（摩擦与反射惯量被放大，N² 律）；折算到关节侧的电机转子惯量 ∝N² ↑ → 抗冲击差、接触"变硬"。腿足机器人偏好小减速比准直驱（QDD）正是为了反驱与力透明度。

</div>
</details>

**Q7 🟡** 摆动相为什么常用摆线（cycloid）足端轨迹？它在离地/触地瞬间的速度特性如何减少打滑与冲击？还有哪些常见摆动曲线？
　🎯 考点：零速触地、加速度连续、Bézier/样条对比。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

摆线参数化下，足端在**离地和触地瞬间水平、竖直速度都为 0**，加速度连续：触地零速 → 不打滑、冲击小；离地零速 → 不蹬擦地面。整条曲线无需分段拼接，一个相位参数 φ∈[0,2π] 解析给出。

替代：Bézier 曲线（MIT Cheetah 用法，控制点可调形状，端点速度可指定）、五次多项式（端点位置/速度/加速度全约束）、正弦抬腿（最简单但触地速度非零）。选择关键看端点边界条件与中段避障高度可调性。

</div>
</details>

## B. 强化学习理论（Q8–Q15）

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q8 🟢** on-policy 与 off-policy 的本质区别？PPO 属于哪类？为什么 locomotion 大量用 PPO 而非 SAC/TD3？
　🎯 考点：数据复用 vs 稳定性、大规模并行采样下 on-policy 的优势。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

on-policy：梯度估计要求数据来自**当前策略**，旧数据用完即弃（PPO 属此类，靠 importance ratio 容忍小幅 off）；off-policy：可重放历史数据（SAC/TD3）。

Locomotion 用 PPO 的原因：GPU 并行仿真使采样近乎免费，on-policy 的"样本贵"劣势被抹平；PPO 超参鲁棒、对 reward 尺度不敏感、无 replay buffer 的稳定性问题，在 4096 env 的吞吐下一晚收敛。SAC 的样本效率优势在仿真里换不来墙钟时间，反而多了温度系数等敏感超参。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q9 🔴** 推导 PPO 的 clipped surrogate objective。`clip_param=0.2` 在做什么？去掉 clip 会怎样？
　🎯 考点：重要性采样比、信任域近似、策略更新幅度限制。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

目标：`L = E[min(r_t Â_t, clip(r_t, 1−ε, 1+ε) Â_t)]`，其中 `r_t = π_θ(a|s)/π_old(a|s)` 是重要性采样比。clip 的作用：当 ratio 已越过 1±ε 且继续移动还能"提高"目标时，**梯度被切断**——即去掉"把策略推得离旧策略很远"的激励，是 TRPO 信任域的一阶近似。min 保证它是真实目标的悲观下界。

去掉 clip：单次 rollout 上做 5 epoch 重复优化会把 ratio 推到极端，策略一步崩坏（性能断崖、再也采不到好数据），即经典的 policy collapse。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q10 🟡** 什么是 GAE？`γ=0.99` 与 `λ=0.95` 各控制什么？它们如何权衡偏差与方差？
　🎯 考点：n-step 回报、TD(λ)、优势估计。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

GAE：`Â_t = Σ_l (γλ)^l δ_{t+l}`，δ 是 TD 误差，相当于对 1-step 到 MC 的所有 n-step 优势做指数加权。

γ=0.99 定折扣视野：有效视野 ≈1/(1−γ)=100 步，50Hz 下约 2 秒——够覆盖一两个步态周期。λ=0.95 调**偏差-方差**：λ→0 全靠 critic 自举（低方差、高偏差，critic 不准时被带偏）；λ→1 接近蒙特卡洛（无偏、高方差）。0.95 是 locomotion 标准折中。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q11 🟡** actor-critic 里 value loss、policy loss、entropy bonus 各起什么作用？`entropy_coef=0.01` 过大/过小分别会怎样？
　🎯 考点：探索、过早收敛、熵正则。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

value loss 把 critic 拟合到回报，给优势估计提供基线（降方差）；policy loss 沿 clipped surrogate 上升；entropy bonus 抬高策略熵、维持探索。

entropy_coef 过大：策略保持高随机不收敛，动作抖、奖励上不去；过小：过早确定性化，钻进局部最优（如原地不动）出不来。0.01 配合可学习 std 是常用平衡点；观察训练中熵曲线应**缓慢单调下降**，骤降说明过早收敛。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q12 🔴** `schedule="adaptive"` + `desired_kl=0.01` 的自适应学习率如何工作？相比固定 LR 为什么更稳？
　🎯 考点：KL 监控、步长自整定。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

每次更新后测 KL(π_old‖π_new)：KL > 2×desired_kl → LR ÷1.5；KL < desired_kl/2 → LR ×1.5（rsl_rl 实现）。本质是把"步长"闭环到**策略实际移动幅度**上而非梯度幅值上。

比固定 LR 稳的原因：奖励尺度、优势归一化状态、训练阶段都会改变梯度量级，固定 LR 在早期可能过大（崩）、后期过小（慢）；KL 自整定让每次更新的策略位移近似恒定，相当于自动信任域。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q13 🟡** `init_noise_std=1.0` 是什么？动作分布为何用对角高斯？训练中探索如何随之演化？std 学成很小意味着什么？
　🎯 考点：随机策略、可学习方差、收敛诊断。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

`init_noise_std=1.0`：对角高斯策略 `π(a|s)=N(μ_θ(s), diag(σ²))` 中 σ 的初值，σ 是**与状态无关的可学习参数**。对角高斯：可重参数化、log-prob 解析、每关节独立方差，简单够用。

演化：早期大 σ 驱动探索，随着确定性好动作被发现，熵正则与 surrogate 的拉锯使 σ 缓慢收缩。σ 学得很小 = 策略已收敛进开发模式；若**过早**变小（奖励还低）就是探索塌缩信号，需调大 entropy_coef 或检查奖励是否惩罚过重。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q14 🟡** `num_steps_per_env=24`、`num_envs=4096`：一次 rollout 收集多少 transition？配合 `num_mini_batches=4`、`num_learning_epochs=5` 共多少次梯度更新？超大 batch 对 PPO 的利弊？
　🎯 考点：batch 规模、更新次数估算、梯度噪声。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

一次 rollout：24 × 4096 = **98,304** transitions。梯度更新：5 epochs × 4 minibatches = **20 次**/迭代，每个 minibatch ≈ 24,576 样本。

超大 batch 利：梯度噪声小、训练稳、GPU 利用率高、优势归一化统计可靠；弊：每样本的更新次数少（数据利用浅，靠 5 epoch 弥补）、丧失小批量噪声的隐式正则、large-batch 下最优 LR 需相应调整。PPO 在 locomotion 里靠"大 batch + 少 epoch + clip"的组合换稳定。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q15 🔴** 为何 locomotion RL 能在数千并行环境、几千万步内收敛？相比真机 RL，sample efficiency 的红利从何而来？这又埋下什么隐患（→ sim2real）？
　🎯 考点：并行采样、仿真加速、分布偏移。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

红利来源：① 仿真比真时快几个数量级且 4096 env **并行**；② reset 免费、摔倒无代价，可以肆意探索；③ 状态可直接读取无需估计；④ 域随机化在仿真里零成本。几千万步在真机上是几个月，在 5090 上是几小时。

</details>

**Q18 🟡** 动作为什么设计成 `default_pos + action_scale·clip(action,±1.5)` 的残差形式，而非直接输出关节绝对角？`action_scale`（髋0.5/大腿小腿0.8145）与 `action_clip` 各起什么作用？

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

**Q19 🔴** position 模式（PD 跟踪 `set_joint_position_target`，Kp=25/Kd=0.5）与 effort 模式（直接力矩）两种动作模式，在可学习性与 sim2real 上各有何取舍？真机上 PD 增益不匹配会怎样？

**Q21 🟡** 终止条件用 `fall_height<0.22` 和 `tipped(proj_grav_z>-0.85≈32°)`。为什么要"早终止"而不是只靠负奖励压制？早终止对回报分布与学习速度有何影响？

**Q22 🟡** `_reset_idx` 在全量 reset 时用 `randint` 随机化 `episode_length_buf`。这一步的目的是什么？不打散 episode 边界会怎样？

<details><summary>📖 参考答案</summary>

`randint` 随机化 `episode_length_buf` 把各 env 的剩余寿命打散，使任意时刻 batch 覆盖 episode 全相位——**rollout 去相关**。

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q23 🟡** 为什么用 `projected_gravity_b` 的 z 分量判翻倒，而非世界系姿态角？为什么"高度阈值"和"姿态阈值"两个终止条件都要保留？

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

`proj_grav_z` 是"重力在机体 z 轴的投影"：与 yaw 无关、无欧拉奇异、对任意倾倒方向统一判据（z > −0.85 即倾角 >≈32°），且就是 obs 里已有的量，sim/real 一致。世界系姿态角需要选转序、有奇异、还得先估姿态。

两条件互补：**只有高度**判不出"趴着但 base 还够高"（如跪倒前倾）；**只有姿态**判不出"姿态端正地塌下去"（腿软下蹲到底、或掉进地形坑）。漏判会让病理行为白拿奖励，所以都保留。

</div>
</details>

**Q24 🟡** 速度跟踪奖励用 `exp(-err/σ)`，`tracking_sigma=1.0` 控制什么？σ 太小为什么会"在 0 附近梯度饱和"，使策略学不动？
　🎯 考点：奖励带宽、梯度信号、塑形几何。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

σ 是 exp 核的"带宽"：误差到多大奖励衰减到 1/e。误差≫σ 时 exp 项 ≈0 且**对误差的导数也 ≈0**——早期策略误差大（站着不动 err≈1），若 σ 太小则跟踪奖励连同梯度一起消失，策略感受不到"往前挪一点会更好"，于是停在 0 奖励平台学不动。

σ 太大则误差 0.2 和 0.8 的奖励差不多，分辨率不足、收敛精度差。经验：σ 取**初始典型误差量级**，本任务 1.0 正好等于目标速度本身。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q25 🔴** 训练初期策略收敛到"原地不动"。本仓库定位根因之一是 `alive=0.5/step` 白送奖励压过了早期微弱的跟踪奖励。为什么 alive/常数存活奖励常诱发 stand-still？有哪些替代设计？
　🎯 考点：局部最优、奖励预算、机会成本。

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

站着每步白拿 0.5；尝试走路则跟踪奖励起初微弱且伴随摔倒风险（摔了连 alive 都没了）。期望回报：**站立 > 早期蹒跚**，策略理性地选择不动——这是奖励预算分配造成的局部最优，不是 bug 是 feature 没设计好。

**Q26 🔴** 策略学出"跳着走"（bound/pronk）。为什么"左右幅度对称"奖励抓不到 bound？本仓库改用**对角相位**（diag1=FL+RR、diag2=FR+RL）配对 + `anti_bound`（同端两脚 z 速度同步惩罚）。解释这两项为何互补、为何能锁死跳跃步态。
　🎯 考点：步态相位、对称性失效模式、关节空间 vs 足端速度判据。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

bound 是左右腿**同步**：左右对称性度量在 bound 下完美对称，该奖励对此病理零灵敏度——对称性奖励只约束"左右镜像"，不约束"对角同相"。

对角配对直接编码 trot 的定义：FL+RR 同相、FR+RL 反相，在**关节空间**惩罚相位偏离；anti_bound 在**足端速度空间**惩罚同端两脚 z 速度同向——这恰是 bound/pronk 的运动学签名。互补性：前者"拉向 trot"，后者"推离 bound"，一个定义吸引子、一个封死最近的逃逸通道；两个空间（关节相位 vs 足端速度）交叉覆盖，钻一个空子会被另一项抓住。

**Q27 🔴** `foot_slip` 惩罚 = Σ(着地脚的水平速度² × 接触掩码)。为什么"脚踩着地却在身下滑动"会被普通速度跟踪奖励放任？接触判定用**高度阈值**近似而非接触力，有什么利弊？

**Q28 🔴** `gait_symmetry` 为什么**故意排除 hip 关节**？（提示：左右 hip 轴镜像 + 合理"对称外撇"会被该公式误判成 bound，导致后腿被锁死贴默认值。）这暴露了奖励设计的什么普遍陷阱？

<details><summary>📖 参考答案</summary>

左右髋的旋转轴是**镜像**的：物理上对称的外撇姿态，两侧髋角**数值符号相反**。gait_symmetry 公式若直接比数值相等，会把"合理对称外撇"误判成不对称/bound 并惩罚——策略发现最便宜的归零方式是把髋钉死在 default 不动，于是后腿髋被锁、步态畸形。排除 hip 后该项只管大腿/小腿的对角相位，髋交给其他项。

**Q29 🟡** `doc/足端打滑问题.md` 推荐 foot air time / contact timing / phase observation 等。foot air time reward 为什么能促成有节律、清晰抬落的步态？

**Q30 🔴** 该文档的核心论断是"**不要只惩罚跳，要让走路更优**"。从优化视角解释：为什么单纯叠加惩罚项常常失败（策略钻空子 / 不敢动 / 代偿）？

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

优化视角：惩罚项只改变损失面局部高度，**不改变可行解集合的拓扑**。策略沿梯度找"代价最小的合规方式"，常见三种失败：① 钻空子——在惩罚公式的盲区里找到新病理（Q26/Q28）；② 不敢动——惩罚总量压过正奖励，最优解退化为保守不动；③ 代偿——用怪异姿态在数学上归零惩罚但物理上更糟。

**Q31 🔴** env 里几乎每个 reward 项都有 `clamp` + `nan_to_num`，`base_height_error` clamp 到 [0,4]。为什么 reward 工程的数值保护如此关键？举例说明一个未 clamp 的平方项如何配合闭链 PhysX 失稳产生 1e28 的单步奖励，并污染整个 batch。
　🎯 考点：数值稳定、异常值传播、训练崩溃定位。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

平方误差项无上界。链条：闭链 anchor 漂移（未开 projection）→ 某 env 单步 root 位置数值发散到 1e14 → base_height 平方误差 ≈1e28 → 单步奖励 1e28。

这一个数进入 batch 后：return/advantage 的均值方差统计被彻底摧毁，归一化后**所有正常样本的优势趋零、异常样本主导梯度**，value function 被打飞，之后若干迭代全是垃圾更新，曲线表现为奖励瞬间尖峰后崩盘。

所以三层防护缺一不可：每项 clamp（限单步贡献）、nan_to_num（断 NaN 传播）、invalid_state 隔离（坏 env 奖励置零 + 强制 reset，见 Q38）——在数据进 PPO **之前**消毒。

</details>

**Q33 🟡** Isaac Sim/PhysX 与 MuJoCo 在接触模型、闭链约束、并行化上的主要区别？本项目为何"Isaac 训练 + MuJoCo sim2sim"两条腿走路？

**Q34 🔴** 平行四连杆膝在 PhysX 里如何处理？为什么 4 个 `*_close_loop` 关节被标 `excludeFromArticulation`（按独立约束求解，不入缩减坐标树）？不开 `enableProjection` 会发生什么（anchor 漂移→1e28 尖峰）？
　🎯 考点：缩减坐标 vs 最大坐标、闭环约束投影、约束漂移。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

PhysX articulation 是缩减坐标**树**，拓扑上不允许环。闭环处理：每个平行四连杆选一根关节标 `excludeFromArticulation`——它退出关节树，作为独立的最大坐标约束（D6/revolute constraint）由迭代求解器单独闭合。

不开 `enableProjection`：迭代求解每步都残留微小约束违反，闭环处误差**无积分反馈地累积**，两个 anchor 渐行渐远 → 约束力随距离暴涨 → 机构突然"爆开"、状态数值发散 → 即 Q31 的 1e28 尖峰源头。projection 在每步求解后把 anchor 几何**硬拉回**重合，相当于给漂移加了位置级复位。仓库用 `projected_loop_joint_names` + `_patch_projected_loop_joints` 启动时统一打补丁。

</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q36 🔴** PhysX 的 `gpu_found_lost_aggregate_pairs_capacity` 等 GPU caps 为什么要随显存调？`2**29` 单项要 8B×2²⁹=4GiB **连续**显存意味着什么？在 8GB/11GB 卡上不调会怎样？
　🎯 考点：宽相位碰撞对、显存预算、连续分配失败。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

这些 caps 是 GPU 宽相位/接触缓冲的**预分配**大小：PhysX 启动时按 cap 一次性 cudaMalloc。`2**29` 项 × 8B = 4GiB 且要求**连续**显存——在 8GB 卡上（系统+模型+渲染已占一截）连续 4GiB 大概率分配失败，启动即 crash 或回退 CPU。

反过来 cap 太小：运行中碰撞对溢出，PhysX 打 warning 并**静默丢弃接触**——表现为脚穿地、机器人掉下平面这类灵异现象。所以 caps 必须按"env 数 × 每 env 碰撞体数"估需求、按显存定上限，仓库里按卡型收紧。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q37 🟡** `solver_position_iteration_count=64` 为什么设这么高？闭链 + 高接触场景对 solver 迭代次数有何要求？迭代太低/太高分别什么后果？
　🎯 考点：约束求解收敛、稳定性 vs 算力。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

闭链约束 + 多点接触构成**刚性耦合的约束系统**，迭代求解器（PGS/TGS）需要足够多次扫掠才能把闭环误差与接触穿透同时压下去。

迭代太低：穿透、四连杆"软掉"变形、anchor 漂移加速（接 Q34 恶性循环）、姿态抖动；太高：求解时间 ∝ 迭代数线性涨，收益边际递减。64 对普通开链偏高，但对"4 个闭环 × 持续足地接触"是必要保险；速度迭代可以远低于位置迭代（位置误差才是闭链命门）。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q38 🔴** 闭链 PhysX 偶发 NaN/Inf。本仓库如何检测（`invalid_state`：root/jpos/jvel 非有限）、如何避免污染 PPO（坏 env 奖励置 0 + `_get_dones` 强制 reset）？为什么绝不能让 NaN 流进 policy/优化器？
　🎯 考点：异常态处理、训练鲁棒性、梯度污染。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

检测：每步对 root pose/vel、jpos、jvel 做 `isfinite` 全检，汇成 `invalid_state` 掩码。处置：该 env 当步**全部奖励置零**（不让坏数据计入回报），`_get_dones` 里强制 done → reset 重置到干净状态；obs 侧再过 nan_to_num 兜底。

绝不能放 NaN 进 PPO：NaN 经网络前向污染输出 → 反传污染梯度 → **Adam 的动量/二阶矩缓存被永久污染** → 下一步起所有权重 NaN，训练不可恢复（不像大数值还能被 clip 救）。哪怕是大有限值也会炸掉优势归一化（Q31）。原则：异常隔离必须发生在数据进 optimizer 之前。

</div>
</details>

## F. Sim2Real 与域随机化（Q39–Q43）

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q39 🟡** 本仓库 `EventCfg` 随机了摩擦 / base 与腿质量 / Kp-Kd / 关节零位 / 周期推搡 / obs 噪声。逐一说明它们各对应真机的哪类不确定性。
　🎯 考点：DR 与真机误差源映射。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

摩擦 0.6–1.4 → 不同地面材质/灰尘/橡胶足垫磨损；base 质量 ± → 电池、外设载荷、配重误差；腿质量 ±20% → 制造公差与线缆；Kp/Kd ±20% → 驱动器固件增益离散化、电机常数个体差、温度漂移；关节零位 ±0.05rad → 装配/编码器标定残差（见 Q42）；周期推搡 ±0.5m/s → 外部碰撞、牵线、地形突变的等效冲击；obs 高斯噪声 → IMU/编码器测量噪声与微小 bias。

每项 DR 都应能指认一个真实误差源，指认不出的随机化只是白白加难度。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q40 🔴** 为什么**训练开 DR、eval 关 DR**（"干净测量"）？DR 过强 / 过弱分别导致什么？如何判断 DR 调到位了？
　🎯 考点：鲁棒性 vs 保守性、评估可比性、域随机化谱。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

训练开 DR：策略必须在物理参数的**分布**上都能活，学到鲁棒（实质是对参数不确定性的 minimax）。eval 关 DR：评估要的是**低方差、可比、可复现**的度量——你想比较两个 checkpoint 的策略优劣，而不是比较它们抽到的物理参数运气；干净测量也方便定位回归。

DR 过强：策略被迫保守（步子小、蹲得低），名义性能下降甚至学不出来；过弱：真机一上去就露馅。调到位的判据：① 各随机参数范围有真机测量依据（real2sim）；② 对 DR 强度做扫描，找性能拐点；③ 终极判据是 sim2sim/真机迁移成功且不过分保守。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q41 🔴** `effort_limit_sim=12` 卡在力矩需求线零余量 → 策略学出"贴力矩上限硬走"（shank 83~84% 时间 ≥90% 上限）→ 真机力矩/电流不足。为什么仿真里**不要**把力矩贴满真实电机峰值（23.7 N·m）？这属于哪类 sim2real gap？
　🎯 考点：执行器极限建模、策略钻空子、保守裕度。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

`effort_limit_sim=12` 恰好压在需求线（峰值≈12N·m）上零余量，策略学到的解是**长时间贴极限输出**（shank 83–84% 时间 ≥90% 上限）——它把"sim 里力矩会被干净地 clip 且无任何后果"当成了可利用的物理。真机上驱动器电流环限幅、母线电压掉压、电机发热降额，根本给不出持续贴限力矩 → 力矩缺口 → 趴下。

也不该贴 23.7N·m 满峰值：数据手册峰值是**秒级瞬态**指标（热与电流约束），策略若学会依赖峰值力矩持续输出，同样真机兑现不了。正解：sim 上限设在"真机可持续兑现"的力矩（留 30–50% 裕量），再加力矩惩罚让策略主动省力。类别：**执行器极限建模**的 sim2real gap（区别于接触/惯量类 gap）。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q42 🟡** 关节零位偏置 ±0.05 rad、Kp/Kd ±20% 为什么是 sim2real 的关键随机项？真机上这些误差来自哪里？
　🎯 考点：编码器零点/标定、驱动器增益离散、温漂。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

零位偏置：真机装配后编码器零点与 URDF 零位之间总有残差（标定精度、键槽/法兰公差、四连杆传动间隙），0.05rad 在 0.3m 腿长上是足端 ~1.5cm 误差——足以改变接触时机与稳定裕度；它**直接平移**"动作→姿态"映射，是最致命又最普遍的误差。

Kp/Kd：驱动器固件定点实现、电机转矩常数个体差异（±10% 常见）、温度对磁性能的漂移，使真实闭环刚度阻尼偏离仿真。两者都属于"策略隐式依赖、却必然不准"的环节，DR 的意义是让策略学到对这两类参数**一阶不敏感**的控制律。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q43 🔴** 策略导出为什么要 TorchScript + ONNX 双格式，并设 onnxruntime vs torch 的 `|Δ|<1e-4` 数值一致性门？数值不一致可能来自哪些环节？
　🎯 考点：算子实现差异、量化/精度、部署可信度。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

TorchScript 供 C++/libtorch 部署链（rl_sar），ONNX 供 onnxruntime/嵌入式推理；双格式还互为**导出正确性的交叉验证**——同一权重两条导出路径结果一致，基本排除导出器 bug。

`|Δ|<1e-4` 门检的是数值一致性，差异可能来自：算子实现差异（gemm 分块顺序、ELU 近似）、fp 累加顺序不同（浮点不结合）、opset 转换语义差（默认 dtype、broadcast 规则）、不支持算子的 fallback 路径、未对齐的输入归一化。1e-4 阈值容忍合理的浮点重排，又能抓住真正的语义偏差；过不了门的模型**禁止上机**——真机上你无法区分"策略不行"和"导出错了"。

</div>
</details>

## G. 状态估计与 HIM（Q44–Q46）

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q44 🔴** 训练 obs 里有 `root_lin_vel_b`，但真机通常**没有** base 线速度的直接传感器。这给 sim2real 埋了什么雷？常见的三类解决思路是什么？
　🎯 考点：可观测性、特权信息泄漏、状态估计。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

雷：训练时 obs 里的 `root_lin_vel_b` 是仿真器直接读的**真值**，真机没有这个传感器——部署要么拿不出这一维，要么用估计值填充，而估计误差（漂移、延迟、接触假设失效时的跳变）是策略从未见过的分布偏移，轻则性能掉、重则摔。

三类思路：① **显式估计**——卡尔曼/互补滤波融合 IMU+腿式里程计得 v̂，并在训练时给该 obs 加上与估计器误差特性匹配的噪声；② **拿掉它**——actor 不用线速度（盲策略），靠历史隐式补偿，即本仓库 HIM 路线；③ **teacher-student/RMA**——特权 teacher 蒸馏给只看可观测历史的 student。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q45 🔴** HIM 用非对称 actor-critic：actor 看 45 维"盲"obs（无 base 线速度），critic 看 51 维特权 obs（+ 真速度 + 扰动）。为什么 critic 可以用特权信息而 actor 不能？部署时用哪个网络？
　🎯 考点：CTDE、价值估计 vs 可部署策略、信息不对称。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

critic 只在**训练时**参与（算优势、降方差），部署时根本不存在，所以它可以看任何特权信息——更准的 value 估计只会让 actor 的梯度更好，不会泄漏到 actor 的输入依赖里。actor 必须只依赖真机传感器可得的 45 维盲 obs，否则不可部署。

这就是 CTDE（集中训练、去中心执行）在单机器人上的用法。部署只导出 **actor**（45 维输入），critic 训练完即弃。

</div>
</details>

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q46 🔴** HIM 如何从本体感知历史隐式估计 base 速度与潜在扰动？它与 RMA（Rapid Motor Adaptation）、teacher-student 蒸馏在思路上有何异同？
　🎯 考点：内部模型、历史编码、隐变量自适应。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

HIM 用本体感知**历史序列**（jpos/jvel/last_action/IMU 的滑窗）编码出隐变量，训练目标显式或对比式地让隐变量**预测 base 速度与等效扰动**，actor 以隐变量为条件——速度被"内部模型"隐式重建而不需要传感器。

对比：RMA 两阶段——先训以真实环境参数（摩擦、质量）为输入的 teacher，再训一个从历史回归出该隐变量的 adaptation module；teacher-student 蒸馏则直接对齐动作分布。HIM 单阶段端到端，估计目标是**状态**（速度/扰动）而非环境参数。共同思想：不可观状态/参数在可观历史中留有签名，用历史编码恢复之；区别只在隐变量定义与训练拓扑。

</div>
</details>

## H. 部署与实时系统（Q47–Q48）

---

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q47 🔴** 从仿真 50Hz 策略到真机实时控制环，会引入哪些 gap：控制频率、通信/计算延迟、CAN 总线、关节符号与零位映射、PD 内环位置？你会用什么"逐级验证"流程（sim → sim2sim → hardware-in-the-loop → 真机）把它们一一隔离？
　🎯 考点：实时性、延迟建模、分层验证。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

gap 清单：① 50Hz 周期抖动（OS 调度）；② 推理 + 总线往返延迟——动作作用在"旧观测"上，等效相位滞后；③ CAN/RS485 带宽：12 电机的命令+反馈调度，丢帧/乱序；④ 关节符号、零位、减速比映射约定 sim vs 固件不一致；⑤ PD 内环跑在驱动器上，频率/离散化/饱和行为与仿真 200Hz 内环不同。

逐级验证：Isaac eval（关 DR 基线）→ MuJoCo sim2sim（隔离"过拟合 PhysX"）→ 导出数值门（隔离导出 bug，Q43）→ 台架 HIL：电机上架空转，跑 `measure_joint_signs.py` 实测符号、校零位、单关节阶跃响应对 Kp/Kd → 吊挂空中踏步（看相位与对称性，零摔倒风险）→ 系绳低速地面行走 → 放绳。每级只引入一类新误差源，挂了就知道是哪类。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q48 🔴【场景】** 一个仿真表现很好的策略，上真机直接趴下/抽搐。给出你的排查优先级与每一步的判据（提示顺序：关节符号/方向 → 零位 → Kp/Kd → 力矩&电流限 → 控制延迟 → DR 不足 → obs 尺度/噪声）。本仓库里哪些工具能帮你？
　🎯 考点：系统化 debug、最小改动定位、工具链。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

排查序（按"杀伤力 × 先验概率"排）：
① **关节符号/方向**：逐关节发小幅偏移看动向（`measure_joint_signs.py`），一个符号反 = 正反馈，瞬间抽搐——别信 URDF 注释，实测；
② **零位**：下发 default 姿态，目测/量角 vs 仿真站姿；
③ **Kp/Kd**：单关节阶跃响应曲线对仿真（超调=Kp 大，软塌=小）；
④ **力矩/电流限与母线电压**：`standup_torque.py` 给出需求线，对驱动器日志看是否饱和/掉压（本机已知病灶，Q41）；
⑤ **控制延迟**：时间戳全链路打点，>1 个控制周期就要在训练中模拟延迟；
⑥ **DR 不足**：前五项都对齐还不稳，回去加 DR 重训；
⑦ **obs 尺度/噪声**：录真机 obs 与仿真分布逐维对比（单位、符号、量纲错很常见）。

前置零成本步骤：先跑 MuJoCo sim2sim——它若也挂，问题在策略/导出而非真机，省一次上机。

</div>
</details>

## J. 模型基控制：MPC / WBC / 经典方法（Q51–Q58）

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q51 🟡** 什么是质心动力学（centroidal dynamics）模型与单刚体（SRBD）近似？四足 MPC 为什么常用 SRBD 而不上全身动力学？这个近似在什么动作下会失效（提示：腿质量占比、甩腿动作）？
　🎯 考点：模型降阶、腿轻假设、模型保真度与求解速度权衡。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **质心动力学**：把全身动力学投影到质心——质心线动量与角动量的变化 = 各接触力/力矩之和 + 重力，6 维方程对任意构型精确成立。
- **SRBD** 进一步假设：① 腿质量可忽略（全身惯量 ≈ 躯干常惯量）；② 姿态变化小到惯量张量近似不变。机器人退化为"一个被几个接触力推着走的刚箱子"。
- **MPC 用它的原因**：状态量小（13 维）、对接触力线性/近线性 → 可凸化为 QP，毫秒级求解撑得起 50–100Hz 重规划；全身动力学非线性非凸，预测域内解不动。
- **失效场景**：腿质量占比大（本机单腿 1.8kg×4 / 整机 11.7kg ≈ 62%——其实不算轻！）、快速甩腿（摆腿角动量不可忽略）、大姿态机动（后空翻）、手臂/负载大幅运动。此时需 centroidal+全身运动学或全身 MPC。

</div>
</details>

**Q52 🔴** 写出凸 MPC（如 MIT Cheetah 风格）的优化问题：状态、决策变量（各支撑足地反力）、动力学约束、摩擦锥约束、代价函数。它为什么是凸的？哪一步把问题凸化了（提示：预先指定接触序列 + 姿态线性化）？
　🎯 考点：QP 形式化、接触序列解耦、线性化。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **状态** `x = [Θ(rpy), p, ω, ṗ, g]`（13 维）；**决策变量**：预测域内各支撑足地反力 `f_i ∈ R³`。
- **动力学约束**：`x_{k+1} = A_k x_k + B_k f_k`——SRBD 在参考 yaw 与参考轨迹处**线性化**后离散化。
- **摩擦约束**：`|f_x| ≤ μ f_z`、`|f_y| ≤ μ f_z`、`f_z ∈ [f_min, f_max]`（线性化摩擦棱锥）。
- **代价**：`Σ ‖x_k − x_ref‖²_Q + ‖f_k‖²_R` → 标准 QP。
- **凸化的两步**：① **接触序列预先由步态调度器指定**（哪只脚何时着地不进优化——去掉了整数变量/互补约束这两大非凸源头）；② 姿态用小角度近似在参考处线性化。剩下的就是线性动力学 + 线性不等式 + 二次代价 = 凸 QP。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q53 🟡** 摩擦锥为什么要线性化成摩擦棱锥（4/8 面体）？保守化带来什么影响？二阶锥规划（SOCP）何时值得？
　🎯 考点：约束凸近似、求解器选型。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- 真摩擦锥 `√(f_x²+f_y²) ≤ μf_z` 是二阶锥约束，QP 求解器不收；用内接 4/8 面棱锥替换后全部约束线性，可用成熟 QP 求解器（qpOASES/OSQP）毫秒级解。
- **保守化**：内接棱锥比真锥小（4 面体最差方向缩水 √2 → 等效 μ 降到 0.71μ），即放弃一部分本可用的切向力，落脚选择更保守；面数越多越贴近真锥但约束数线性涨。
- **SOCP 值得的场合**：低摩擦面（冰面，μ 裕量本来就小，保守化损失致命）、需要精确摩擦边界的操作任务；代价是求解器更重（ECOS/Mosek），实时预算要重新核。

</div>
</details>

**Q54 🔴** ZMP 判据与接触力矩锥（CWC）判据的关系与各自适用范围？四足在非共面接触（爬台阶）时为什么 ZMP 失效？
　🎯 考点：动平衡判据、平面假设、广义稳定裕度。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **ZMP**：地面上净接触力矩水平分量为零的点；"ZMP ∈ 支撑多边形"是动平衡判据。**隐含假设**：所有接触点**共面**且摩擦充分（只查压力中心、不查切向）。
- **CWC**：所有接触能产生的六维力旋量的凸锥；"重力+惯性力旋量 ∈ CWC"是**广义判据**——任意接触几何、含摩擦约束，ZMP 是它在"平面接触+无限摩擦"下的特例。
- **台阶上 ZMP 失效**：接触点不共面时"支撑多边形"无良定义——各足压力作用面不同高度，水平投影多边形不再对应可行力矩集合；打滑约束也完全没被检查。正确做法是用 CWC 或逐接触摩擦锥的可行性检验（即 MPC 里的约束形式）。

</div>
</details>

**Q55 🔴** 解释 WBC（全身控制）的任务空间分层：动力学一致的零空间投影或加权 QP 如何同时满足"接触约束 > 姿态任务 > 摆腿任务"的优先级？与 RL 策略输出关节目标相比，WBC 的优势与脆弱点各是什么？
　🎯 考点：层级 QP、零空间、模型基 vs 学习基。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **严格分层（零空间投影）**：最高优先级任务先解（接触点不动、动力学一致），次级任务只在前级解的**零空间**里找最优：`q̈₂ = q̈₁ + N₁ z`，N₁ 是前级任务雅可比的零空间投影；逐级嵌套保证低优先级**数学上不可能**破坏高优先级。
- **加权 QP**：所有任务进一个 QP 按权重折中，软优先级，数值更稳但"优先级"只是近似。层级 QP（HQP）则逐级解一串 QP。
- **WBC 优势**：力/姿态约束**显式可证**满足（接触不滑、力矩限不破）、行为可解释可调、不需训练。**脆弱点**：依赖模型精度（质量/惯量/接触点位置错则力分配错）、接触状态估计错时整层失效、对未建模扰动（软地面）没有学习类方法那种"见过就会"的鲁棒；状态机切换处易抖。
- RL 输出关节目标则相反：不需精确模型、对见过的扰动鲁棒，但无硬约束保证、分布外行为不可预测。

</div>
</details>

**Q56 🟡** 经典"Raibert heuristic"落脚点公式 `p = p_hip + (T_stance/2)·v + k·(v − v_cmd)` 各项的物理意义？为什么这么简单的规则能稳定动态步态？RL 策略里有没有它的影子？
　🎯 考点：落脚点调节、速度反馈、捕获点直觉。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **第一项** `p_hip`：落脚基准在髋正下方。**第二项** `(T_stance/2)·v`：支撑相内身体要前移 `T_stance·v`，落脚提前半个支撑期，使支撑段对称——净加速为零的"中性落脚点"。**第三项** `k(v − v_cmd)`：速度误差反馈——跑快了把脚往前多放（落脚在质心前方产生减速力矩），跑慢了少放。
- **为什么能稳**：把"身体平衡"问题转化为"落脚位置"的离散反馈控制，每步是一次采样控制；倒立摆动力学下落脚点对速度的影响近似线性，简单 P 控制即可镇定——这正是捕获点思想的雏形。
- **RL 的影子**：训练好的策略可视化常能看到同款行为——速度偏快时足端落点系统性前移、被推后立即朝推力方向跨步。RL 等于隐式重新发现了 Raibert 律（可作为策略合理性的"体检项"）。

</div>
</details>

**Q57 🔴** 什么是捕获点 / DCM（发散运动分量）？把 LIP 模型的不稳定模态分离出来对推搡恢复有什么用？四足比双足在这个问题上"容易"在哪里？
　🎯 考点：LIP、不稳定子空间、支撑多边形。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- LIP（线性倒立摆）动力学 `ẍ = ω²(x − p)`（ω=√(g/h)）有一稳一不稳两个模态。**DCM** `ξ = x + ẋ/ω` 恰是不稳定模态坐标：它满足一阶不稳定动力学 `ξ̇ = ω(ξ − p)`，而质心会自动收敛跟随 DCM。**捕获点** = 当前 DCM 的位置：脚踩到那里则系统能停下（DCM 不动点）。
- **用处**：平衡控制从"控质心二阶系统"降为"只镇定一阶不稳定量 ξ"——推搡后 ξ 瞬间外移，控制律只需把落脚点/ZMP 安排在 ξ 外侧即可恢复，给出**何时必须跨步**（ξ 出支撑多边形）的解析判据。
- **四足更容易**：支撑多边形大得多（4/3/2 足支撑 vs 双足一只脚），ξ 出界的余量大；且任意时刻可用对角双足支撑换步，"跨步恢复"选项多、时间窗宽。双足 ξ 一出脚掌就必须精确跨步，容错极小。

</div>
</details>

**Q58 🔴** MPC+WBC 栈与端到端 RL 各自的失效模式是什么（模型失配 vs 分布外）？讨论"RL 出步态意图 + WBC 出力矩"或"RL 调 MPC 参数"这类混合架构的动机与代价。
　🎯 考点：混合架构、模块化边界、失效模式互补。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **MPC+WBC 失效于模型失配**：质量/摩擦/接触点估计错、未建模柔性、软地面——模型错则最优解错，且错得自信。**RL 失效于分布外**：训练分布没覆盖的地形/扰动/执行器状态，行为不可预测，且无法在线"解释"。两者失效模式近乎互补。
- **RL 出意图 + WBC 出力矩**：RL 在慢时间尺度选落脚点/步态参数/质心目标（吃分布外鲁棒与长视野），WBC 在快时间尺度兑现并**强制满足**摩擦锥/力矩限/接触约束（吃硬约束保证）——RL 的"无保证"被 WBC 兜底，WBC 的"模型死板"被 RL 的目标自适应缓解。
- **RL 调 MPC 参数**：学步态周期/增益/代价权重的调度器，保留 MPC 主体可解释性。
- **代价**：接口设计成为新的失效面（RL 给出 WBC 不可行的意图怎么办）、两套时序的工程复杂度、训练时必须把下层控制器放进环（训练慢且下层一改 RL 要重训）、调试跨两个范式。

## K. 进阶 RL 训练技术（Q59–Q66）

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q59 🟡** 什么是课程学习（curriculum）？给本任务设计速度指令课程与地形课程（提示：本仓库 terrain 已有 `MeshPlaneTerrainCfg(proportion=0.2)` 等生成器配置），自动课程（按成功率升降级）怎么实现？
　🎯 考点：渐进难度、自动课程、env 分桶。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **课程学习**：从易到难安排训练分布，让早期策略在"够得着"的任务上拿到梯度，再逐步逼近目标难度——本质是对探索问题的先验调度。
- **速度课程**：指令范围从 ±0.3 m/s 起步，策略跟踪误差 < 阈值后扩到 ±0.6 → ±1.0 → 加侧移/转向；始终保留 10–20% 零指令样本（站立不退化）。
- **地形课程**：本仓库 `--terrain curriculum` 已有骨架——terrain generator 按行排难度（平地 → 粗糙 → 楼梯），所有 env 从第 0 行起步。
- **自动课程**：每 env 记录在当前难度的成功率（如走完 episode 且跟踪达标）；高于上阈值 → 该 env 升一级地形/扩指令范围，低于下阈值 → 降级。Isaac Lab terrain curriculum 的"走得远就升行"就是这个机制；要点是**按 env 分桶独立升降**，保持 batch 内难度混合。

</div>
</details>

**Q60 🔴** AMP（Adversarial Motion Priors）如何用判别器把动捕/动物数据的"风格"注入奖励？它解决了手工奖励的什么痛点？训练不稳的常见原因？
　🎯 考点：对抗模仿、风格奖励、GAN 失稳。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **机制**：判别器 D 学着区分"参考数据的状态转移 (s,s′)"与"策略产生的转移"；策略的奖励 = 任务奖励 + **风格奖励**（D 认为你像参考数据的程度，如 `−log(1−D)`）。策略被推向参考运动的**分布**，而非逐帧跟踪某条轨迹。
- **解决的痛点**：步态自然度（关节协调、节律、躯干姿态）几乎无法用手工项写全——Q24–Q32 那一堆对症奖励，AMP 用一个判别器全包了；且不需要相位对齐/重定向到逐帧参考。
- **失稳来源（GAN 通病）**：① 判别器过强 → 风格奖励梯度消失（需梯度惩罚/谱归一化、限制 D 更新频率）；② 模式塌缩 → 只学参考数据的一种步态；③ 任务奖励与风格奖励权重失衡 → 要么"好看但不走"要么"能走但难看"；④ 参考数据与机器人形态差异大时重定向误差被判别器放大。

</div>
</details>

**Q61 🟡** 利用机器人左右对称性做数据增广（镜像 obs/action 对）或对称性损失，为什么能加速训练并改善步态对称？实现时最容易错在哪（提示：Q28 的镜像符号问题在这里再次出现）？
　🎯 考点：群对称、数据增广、关节映射表。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- 行走任务对左右镜像近似等变：把每条经验 (s,a,r,s′) 镜像成 (Ms, Ma, r, Ms′) 再加入 batch，样本量翻倍且**强制策略在镜像对上行为一致**——左右腿不再各学各的，收敛快、步态对称性免费提升。另一路线是对称性损失：`‖π(Ms) − Mπ(s)‖²` 作为正则。
- **最容易错的地方**（Q28 的镜像坑重现）：镜像矩阵 M 的**逐维符号表**。obs 里 y 向速度、roll、yaw 角速度、重力 y 分量要翻号；关节要做"左右互换 + 按各自旋向定符号"——左右 hip 轴本身镜像安装，hip 角互换后**还要再翻号**，而 thigh/shank 可能不翻。一张表错一个符号，增广就在系统性教错的数据上训练，比不增广更糟。**验证法**：取一个状态，镜像两次应恢复原状；用 FK 检查镜像后关节角对应的足端位置确实是镜像点。

</div>
</details>

**Q62 🔴** 帧堆叠（obs 历史拼接）vs LSTM/GRU 循环策略：在部分可观测（无线速度、有延迟）下各自的优劣？rsl_rl 的 recurrent 实现里 hidden state 在 reset 时要注意什么？
　🎯 考点：POMDP、记忆机制、BPTT 与 hidden 重置。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **帧堆叠**（拼 5–50 帧历史）：实现简单、训练就是普通前馈 PPO、推理零状态（部署友好、热更新易）；劣势是记忆窗口硬截断、obs 维度线性涨、对长时依赖（慢时变的负载/温度）无能为力。
- **LSTM/GRU**：理论记忆无限长、参数效率高、能积分出慢变量（HIM 类估计天然合适）；劣势是要 BPTT（rsl_rl 按时序段训练）、部署要维护 hidden state、调试难（hidden 不可解释）。
- **reset 注意点**：① env reset 时该 env 的 hidden 必须**清零**，否则上一条 episode 的记忆泄漏进新 episode（学到依赖"出生前"信息的假象）；② rollout 存储要把 hidden 随轨迹一起存，minibatch 重放从正确的 hidden 起点展开；③ 部署侧 reset/急停后同样要清 hidden，且策略热切换时 hidden 没有可迁移语义（Q85）。

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q63 🟡** 优势归一化、观测归一化（running mean/std）、奖励缩放三者分别解决什么问题？部署时观测归一化的统计量怎么处理（提示：必须随模型一起导出冻结）？
　🎯 考点：数值条件、归一化状态的持久化。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **优势归一化**（batch 内减均值除 std）：让 policy 梯度的尺度与奖励绝对量级解耦，clip/LR 超参跨任务可复用；也压制单样本大优势的主导（但救不了 1e28 那种极端，见 Q31）。
- **观测归一化**（running mean/std）：把量纲悬殊的各维 obs（rad vs rad/s vs 无量纲）拉到同方差，改善网络输入的条件数，加速收敛。
- **奖励缩放**：控制 return 的绝对量级 → 影响 value loss 的尺度与 value/policy 梯度平衡。
- **部署关键**：running 统计量是**模型的一部分**——导出时必须随权重一起**冻结**进 TorchScript/ONNX 图（或显式导出 mean/std 常量）；部署侧用自己另算的统计量 = 喂给网络一个平移缩放过的异世界，必挂。这也是 Q43 数值一致性门要连归一化一起测的原因。

**Q64 🔴** 训练曲线诊断：给出 (a) 奖励上升但 episode 长度下降；(b) 熵骤降奖励停滞；(c) value loss 爆炸；(d) KL 持续顶 desired_kl 上限，四种症状各自的可能病因与处理。
　🎯 考点：训练病理学、指标交叉验证。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **(a) 奖励↑ 长度↓**：策略找到"短命高分"路径——单步奖励密度被刷高（如冲刺拿跟踪分然后摔），或终止惩罚不足。查分项 `_episode_sums`：哪项在涨；对策：把存活纳入回报结构（终止罚款）、检查正奖励是否可被自杀式刷分。
- **(b) 熵骤降奖励停滞**：过早确定性化困在局部最优（常见：原地不动）。对策：加大 entropy_coef、检查是否有 alive 类白送奖励（Q25）、early-stage 扩大探索（init_noise_std）、奖励塑形让早期有梯度（Q24）。
- **(c) value loss 爆炸**：回报里进了异常值（Q31 的尖峰）、奖励尺度突变（改了权重没重训 critic）、或 γ 下回报量级远超网络输出范围。对策：先查 reward 分项有无尖峰/NaN，再看是否需要 value clip / 奖励缩放。
- **(d) KL 顶上限**：每次更新策略都被推得太远——LR 偏大、优势异常值、或 minibatch 间数据分布差异大。自适应 LR 会自动压（Q12），若 LR 已被压到很小还顶 KL，多半是优势计算有异常样本，回到 (c) 的排查。

**Q65 🟡** 随机种子对 PPO locomotion 结果的影响有多大？报告实验结果时该怎么做才算严谨（多种子、置信区间）？为什么"单种子曲线对比"会误导决策？

**Q66 🔴** 把本任务从"固定前进"升级为"全向速度跟踪 + 原地转向 + 站立"多指令策略（接 Q17），训练分布如何设计才能避免"平均而平庸"？零指令（站立）样本比例过高/过低各导致什么？

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **分布设计**：① 指令空间分桶采样（前进/后退/侧移/纯转向/混合/零指令），保证每个 batch 各桶都有覆盖，而非均匀采样让"小速度混合指令"占满；② 难度课程（Q59）从小范围扩张；③ episode 中途重采样指令，强迫学**过渡**（加减速、走→停）；④ 跟踪奖励的 σ 按指令幅度归一，否则大速度指令的误差天然大、奖励天然低，策略会偏科。

## L. 显式状态估计与接触估计（Q67–Q71）

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q67 🟡** 设计一个腿式里程计：IMU 预测 + "支撑足速度为零"假设做更新的卡尔曼滤波。写出状态向量、预测方程、更新方程的核心结构。打滑时这个假设怎么破坏估计？
　🎯 考点：KF 结构、零速更新、外点剔除。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **状态**：`x = [p, v, q(姿态), b_a, b_g]`（位置/速度/姿态/加计 bias/陀螺 bias），可扩展各支撑足位置 `p_foot,i`。
- **预测**（IMU 高频积分）：`v̇ = R(q)(a_m − b_a) + g`，`ṗ = v`，`q̇ = q ⊗ (ω_m − b_g)`；协方差按 IMU 噪声传播。
- **更新**（每个支撑足）：零速假设 → 足端世界速度为 0，即 `v + R(ω×) p_foot^b + R J q̇ ≈ 0`，由关节角/角速度经 FK/雅可比给出"以足为锚的 base 速度"伪测量；或等价地用"支撑期足位置不变"做位置差更新。
- **打滑破坏**：足在动而滤波器以为它静止 → 伪测量系统性偏移，速度估计被拖向错误值且协方差还自信地收缩。**对策**：打滑检测（测量残差卡方检验做外点剔除）、按接触置信度调测量噪声 R、滑则剔除该足更新。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q68 🔴** IMU 的加速度计 bias 与陀螺 bias 对速度/姿态估计各造成什么样的漂移（一次积分 vs 二次积分）？为什么 bias 要放进滤波器状态在线估计？
　🎯 考点：误差传播、可观测性条件。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **加计 bias**：一次积分进速度 → 速度误差 ∝ `b_a·t` 线性漂；二次积分进位置 → ∝ `t²` 平方漂。
- **陀螺 bias**：先漂姿态（∝ `b_g·t`），姿态错导致**重力补偿错**——`g·sin(Δθ) ≈ g·b_g·t` 被当成真实加速度积进速度 → 速度误差 ∝ `t²`、位置 ∝ `t³`。所以陀螺 bias 杀伤力更大。
- **必须在线估计**：bias 随温度/上电缓变，出厂标定保不住；好在它们**可观**——零速更新（支撑足）与重力方向观测能把 bias 与真运动区分开，放进状态让 KF 自动收敛；不估则上面的多项式漂移全额兑现。这正是腿式机器人比无人机幸运的地方：每一步触地都是一次"准零速"校准机会。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q69 🟡** 不装足底力传感器，如何从关节力矩残差 / 电流 / 广义动量观测器检测足端接触？各方法的延迟与误检特性？
　🎯 考点：动量观测器、外力估计、接触检测。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **力矩残差法**：逆动力学算"无接触应需力矩" `τ_model`，与实际（电流×Kt 折算）比，残差大 = 有外力。简单但要好的动力学模型，且加速度项噪声大。
- **广义动量观测器**：`r = K(∫(τ − C^T q̇ + G − r)dt − M q̇)`，r 收敛到外力矩等效——**不需要加速度测量**（避开数值微分噪声），是工业标准做法；延迟由观测器带宽 K 定（典型几个 ms 到几十 ms），K 大快但噪声敏感。
- **电流阈值法**：支撑相膝/髋电流显著高于摆动相，阈值判断；最简单、延迟最低，但受步态/负载/温度影响阈值难定，误检多。
- **共性权衡**：带宽（延迟）vs 误检率；触地冲击瞬间最容易检测，**离地时刻**（力渐变到零）反而模糊。工程上常融合多源（电流 + 高度 + 步态相位先验），本仓库 RL 侧的高度阈值近似（Q27）是零传感器要求的最低配版本。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q70 🔴** 对比：把估计器与策略分开（估计→obs→策略）vs HIM 式端到端隐式估计。哪种对估计器故障更鲁棒？哪种更好调试？真机出问题时你怎么判断是估计坏了还是策略坏了？
　🎯 考点：模块化 vs 端到端、可观测性调试。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **分开（显式）**：估计量有物理语义，可单独验证（对动捕真值测 RMSE）、单独升级；但策略**隐式依赖估计误差特性**——估计器换版本/故障模式变了，策略等于遇到分布偏移，鲁棒性取决于训练时注入的误差模型像不像（Q71）。
- **HIM（隐式）**：没有"估计器坏"这个独立故障模式（估计与控制一体优化，对自身估计不确定性能自适应地保守）；代价是隐变量不可解释、出问题只能整体回归测试，且换传感器布局要重训。
- **真机定责流程**：① 显式系统——录估计输出对照离线真值（动捕/慢速场景下里程计自洽性）：估计准而行为差 → 策略问题；估计漂 → 估计器问题。② HIM——回 MuJoCo 重放真机 obs 序列看动作是否合理；用辅助解码头（训练时顺手加的速度预测头）输出"隐式估计的速度"作为可观测探针。**实践建议**：即使走 HIM 路线，也保留一个并行运行的显式估计器做监控对照，不进控制环。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q71 🔴** 训练时如何把"真机估计器的误差特性"注入仿真（在真值上叠加有色噪声/延迟/打滑跳变，而非白噪声）？为什么白噪声 DR 不足以覆盖估计器误差？
　🎯 考点：噪声建模保真度、误差谱。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **白噪声不够的原因**：估计器误差是**有色、有结构**的——漂移（低频随机游走）、延迟（确定性相位滞后）、打滑跳变（与步态事件相关的突发偏移）、bias（常值段）。白噪声在频谱上是平的，策略很容易学会"平均掉"零均值高频噪声，但对低频漂移与相关性误差毫无准备——这两者恰恰最伤。
- **注入方法**：① **随机游走/一阶马尔可夫（OU 过程）噪声**模拟漂移：`n_{t+1} = α n_t + w_t`；② **延迟队列**：obs 延迟 1–3 个控制周期随机；③ **事件相关误差**：触地瞬间给速度估计加脉冲偏移（模拟打滑/冲击下估计器抖动）；④ episode 级随机 bias（每条 episode 抽一个常值偏移）。
- **黄金标准**：把真实估计器**放进仿真环里跑**（仿真出 IMU/编码器原始量 → 真估计器代码 → 估计值进 obs），误差特性自动 100% 保真；代价是仿真吞吐下降。退一步是录真机估计误差序列、拟合其谱特性再回放。

## M. 执行器与硬件深入（Q72–Q76）

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q72 🔴** ETH ANYmal 的 actuator network 用什么输入预测什么输出？它弥补了理想力矩源假设的哪些缺陷（SEA 弹性、摩擦、迟滞、带宽）？本仓库的 GO-M8010-6 小减速比方案需要 actuator net 吗，为什么？
　🎯 考点：执行器建模、SEA vs QDD、模型必要性判断。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **ANYmal actuator net**：输入 = 位置误差与关节速度的**短历史**（若干帧），输出 = 实际输出力矩；用真机采集的(指令, 状态)→力矩数据监督训练，替换仿真中的理想执行器。
- **弥补的缺陷**：ANYdrive 是**串联弹性执行器（SEA）+ 大减速比谐波**——弹性元件动力学、谐波摩擦/迟滞、电流环带宽限制，使"指令力矩→实际力矩"是个有记忆的非线性动态系统，理想力矩源假设误差巨大。
- **本仓库的判断**：GO-M8010-6 是 N=6.33 的准直驱——无弹性元件、行星减速摩擦小、力矩传递接近线性，"理想执行器 + T-N 包络限幅 + Kp/Kd DR"已covers大部分误差（仓库已把 T-N 曲线按 N 反射建模进仿真）。**结论：不需要完整 actuator net**；更值得做的是②延迟建模与②摩擦补偿两个轻量项。判断准则：执行器动态的"非理想程度"（弹性/迟滞/带宽）超出简单模型能描述时才上数据驱动建模。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q73 🟡** 电机热模型：铜损 ∝ I²、热时间常数、连续/峰值力矩的本质区别。给定 trot 步态下 shank 关节力矩时间序列，怎么估算电机温升并判断能否持续运行？
　🎯 考点：RMS 力矩、热等效、占空比。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **本质**：发热 `P_cu = I²R ∝ τ_rotor²`——峰值力矩受电流/磁饱和限制（秒级），连续力矩受**散热平衡**限制（`τ_rms` 对应的铜损 = 最大可持续散热功率）。两者之间靠热时间常数（电机绕组典型几十秒～分钟级）缓冲。
- **估算流程**：① 力矩序列折算转子侧 `τ_rotor = τ_joint/(N·η)`，电流 `I = τ_rotor/Kt`；② 取 **RMS**（不是均值！发热看平方）：`I_rms² R` = 平均发热功率；③ 一阶热模型 `T(t) = T_amb + P·R_th(1 − e^{−t/τ_th})`，稳态温升 = `P·R_th`；④ 判据：稳态温升 + 环境温度 < 绕组绝缘等级（且注意 Kt 随温度下降 → 同力矩要更大电流的正反馈）。
- **快速判断**：`τ_rms < τ_rated`（额定即连续）则可无限跑；超出则按热时间常数算可运行时长。本仓库 `standup_torque.py` 输出的力矩 CSV 正可直接做这个分析——这也解释了 Q97 的"跑 10 分钟趴下"模式。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q74 🟡** 准直驱（QDD）、谐波减速、行星减速 + SEA 三种关节方案在力控带宽、反驱性、抗冲击、成本上的对比？为什么足式机器人主流走向 QDD？
　🎯 考点：传动方案权衡、接触任务的需求。

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

**Q75 🔴** 驱动器的"力矩控制模式"内部其实是电流环（FOC）。从策略输出的 τ 到实际输出力矩之间隔着哪些环节（τ→iq 标定、电流环带宽、磁饱和、齿槽转矩、减速器效率与摩擦）？哪些可以标定补偿、哪些只能靠 DR？

**Q76 🟡** 母线电压从 24V 掉到 20V 时，电机的 T-N 包络如何变化？为什么"高速段力矩缺失"先于"低速段"出现？这与本仓库"真机力矩不足"的诊断（Q41）如何相互印证？
　🎯 考点：反电动势、电压裕度、包络收缩。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- 电机电压方程：`U = I·R + Kₑ·ω`。可用力矩 `τ ∝ I = (U − Kₑω)/R`——**T-N 包络右上角随 U 线性内缩**：空载最高转速 `ω_max = U/Kₑ` 直接降 ~17%（24→20V），每个转速下的可用力矩同步下降。
- **高速段先缺力矩**：低速时反电动势 `Kₑω` 小，电压裕量大，电流（力矩）仍能拉满——限制是电流上限；高速时 `Kₑω` 吃掉大部分母线电压，留给电流的 `U − Kₑω` 所剩无几——限制变成电压。所以电压跌时**恒力矩区缩短、恒功率区下移**，高速段最先露馅。
- **与本仓库诊断互证**：站立撑起/蹬地是"中速 + 大力矩"工况，恰在包络拐点附近；电池内阻在大电流下掉压 → 包络瞬时收缩 → 实际力矩达不到指令——与"撑起吃力、站姿下垂"症状一致。**判别实验**：监控大负载瞬间的母线电压（掉压幅度）与电流（是否被驱动器限幅截顶），区分"电压不够"还是"电流上限设低"（todo 力矩专项的两条排查线）。

**Q77 🟡** MuJoCo 的 `solref`/`solimp` 如何参数化接触软硬？把接触调"硬"对 timestep 与稳定性有什么要求？与 PhysX 的接触刚度概念怎么对应着调，才能让 sim2sim 两边行为接近？
　🎯 考点：软约束参数、跨仿真器对齐。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- MuJoCo 把所有约束（含接触）建成**软约束**：违反量按弹簧-阻尼动力学衰减。`solref=(timeconst, dampratio)` 给恢复时间常数与阻尼比（临界阻尼=1）；`solimp` 定约束"阻抗"随违反深度的过渡曲线（d0→d_width 之间从软到硬）。
- **调硬的要求**：`timeconst ≥ 2·dt` 是稳定底线——想要 2ms 的硬接触就得 ≤1ms 的步长。本仓库 `play_mujoco.py` 用 1000Hz 物理就是为了闭链 `<connect>` 等式够刚（同一机制）。
- **与 PhysX 对齐**：PhysX 接触是冲量求解 + `contact_offset/rest_offset` + 材料恢复系数/摩擦，没有直接对应参数。**实操对齐法**：不对参数、对**行为**——做标定实验（同高度落体的回弹高度、同载荷下的穿透深度、斜面起滑角），在两边调到曲线一致。这正是 sim2sim 的预备工作（Q79）。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q78 🟡** MuJoCo 用 `equality/connect` 约束闭链与 PhysX 的 excludeFromArticulation+projection 路线有何本质区别？为什么 MuJoCo 闭链通常更稳？
　🎯 考点：广义坐标软约束 vs 最大坐标迭代投影。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **MuJoCo**：树用广义坐标，环用 `connect` 软等式约束，与接触约束**在同一个凸优化问题里联立求解**——约束力来自全局一致的最优解，违反量按 solref 指数衰减，**自带"弹性回中"，不会累积漂移**。
- **PhysX**：被排除的环关节是独立的最大坐标约束，与 articulation 由迭代求解器**分别/交替**处理；每步残差没有内在回中机制，靠 projection 事后几何硬拉（Q34）——拉的本身又可能与接触约束打架。
- **MuJoCo 更稳的根源**：① 统一求解器全局一致 vs 分块迭代局部修补；② 软约束天然有回复动力学 vs 硬约束残差累积；③ 广义坐标下环闭合误差只有 3–6 维 vs 最大坐标下两刚体 12 维状态都可漂。代价是软约束有微小"呼吸"形变、吞吐低——所以 Isaac 训练、MuJoCo 验证（Q33）。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q79 🔴** sim2sim 时发现：同一策略 Isaac 里步态正常、MuJoCo 里足端轻微打滑且 base 高度低 1cm。列出至少 5 个可能的差异源及逐一排查方法（接触参数、摩擦模型、PD 实现、积分器、质量/惯量导入差异、闭链处理）。
　🎯 考点：跨仿真器差异定位、控制变量。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

（每项都给"隔离实验"）

1. **接触软硬**（最可疑——"高度低 1cm"是软约束被压出的形变签名）：静置机器人对比两边穿透深度/平衡高度；调 solref 变硬或减 timestep 看高度差是否缩小。
2. **摩擦模型**：PhysX 锥面近似/各向异性 vs MuJoCo 椭圆锥；斜面起滑角实验对齐等效 μ；打滑量随 μ 扫描曲线对比。
3. **PD 实现**：PhysX 关节驱动是隐式积分进求解器（同 Kp 下更稳更"硬"），MuJoCo position actuator 是显式力；单关节阶跃响应对比，必要时 MuJoCo 侧手写显式 PD 对齐采样时序。
4. **质量/惯量导入**：USD→MJCF 转换可能丢/错惯量与 CoM（`tools/asset/usd_to_mjcf.py` 路径）；逐 body 打印两边 mass/inertia/CoM diff，零容忍。
5. **闭链处理**（Q78）：MuJoCo 软 connect 让四连杆微"软"，等效腿长缩短 → 直接贡献 base 高度差；加大 connect 刚度/减步长看响应。
6. **积分器与步长**：200Hz PhysX TGS vs 1000Hz MuJoCo 半隐式欧拉；统一控制频率边界条件再比。
- **方法论**：一次只对齐一项（控制变量），从静态（站立高度）→ 准静态（缓慢下蹲）→ 动态（行走）逐级对比，先解释静态差异再看动态。

</details>

**Q80 🟡** 显式欧拉、半隐式欧拉、RK4 在刚性接触系统里的稳定性差异？为什么物理引擎几乎都用半隐式欧拉 + 小步长而不是 RK4 + 大步长？
　🎯 考点：刚性系统、积分器稳定域。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **显式欧拉**：稳定域不含大负实特征值——刚性弹簧（接触）直接发散，不可用。**半隐式（symplectic）欧拉**：先更新速度再用新速度更新位置，对振荡/接触系统稳定域大得多，且辛性质保证能量长期不漂，每步只算一次力。**RK4**：光滑系统下 4 阶精度，但每步 4 次求值，且稳定域对刚性问题仍有限。
- **关键：接触不是光滑 ODE**。碰撞/约束是不连续事件（LCP/互补问题），RK4 的高阶精度建立在导数连续的泰勒展开上——跨过接触瞬间的中间求值点毫无意义，高阶优势直接报废。
- 物理引擎的真实范式是"**离散时间冲量求解**"：每步求解一次约束冲量再积分，半隐式欧拉与这个框架天然契合；精度靠小步长买（还顺带满足软约束 timeconst ≥ 2dt 的需求，Q77），比在不连续系统上浪费高阶求值划算得多。

**Q81 🟡** Linux 上跑 50Hz 控制环：PREEMPT_RT、SCHED_FIFO、CPU 隔离、内存锁定各解决什么问题？怎么测量并报告控制环的抖动（jitter）分布？
　🎯 考点：实时性保障、延迟测量。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **PREEMPT_RT** 内核补丁：让内核态几乎处处可抢占，把"最坏情况内核延迟"从毫秒级压到几十微秒——治"内核正忙不理你"。
- **SCHED_FIFO** 实时调度类：控制线程优先级高于一切普通进程，就绪即运行——治"被别的进程挤掉"。
- **CPU 隔离**（isolcpus/cset + IRQ affinity）：留一个核只跑控制线程，无邻居、无中断风暴——治缓存污染与突发抢占。
- **内存锁定**（mlockall + 预触碰）：禁止缺页换出——治"运行中突然缺页卡 10ms"。
- **测量**：控制环里每拍 `clock_gettime(CLOCK_MONOTONIC)` 记实际唤醒/完成时刻，跑 ≥30 分钟（覆盖慢事件），报告**直方图与 P99/P99.9/最大值**——实时系统看尾延迟不看均值；附加压力测试（同机跑 stress-ng）下的同款直方图。50Hz/20ms 周期下，抖动预算一般要压到 <1ms。

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q82 🟡** 12 个电机挂在 RS485/CAN 总线上：一拍内"发 12 命令 + 收 12 反馈"的总线时间预算怎么算？分组多总线、提高波特率、命令反馈合包各是什么思路？
　🎯 考点：总线带宽、通信调度。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **预算公式**：单帧时间 = 帧位数/波特率。以 RS485 4Mbps、宇树 GO 协议（命令 17B、反馈 16B ≈ 170/160 bit 含起停位）为例：一问一答 ≈ (170+160)/4M ≈ 82.5μs，加协议间隔取 ~100μs；12 电机串行轮询 ≈ 1.2ms。20ms 周期下占 6%——可行，但若波特率低一档（CAN 1Mbps：每对 ~260μs×2，12 电机 ≈ 6ms+）就开始挤压推理预算。
- **三个优化思路**：① **分组多总线**（本仓库正是 4 路 USB-RS485，每路 3 电机）——总线级并行，轮询时间 ÷4，还隔离故障域；② **提高波特率**——线性缩时间，受线长/拓扑/收发器限制；③ **命令反馈合包/广播**——一帧带多电机命令、反馈按时隙应答，省掉每对的帧头开销与方向切换时间（RS485 半双工换向是隐藏大头）。
- 还要算上 **USB 的坑**：USB-串口适配器的批量传输调度引入 0.1–1ms 级抖动，硬实时场合宁可用原生串口/CAN 控制器。

</div>
</details>

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q83 🔴** 设计真机安全层（独立于策略）：关节位置/速度/力矩软限位、姿态超限自动卸力、通信超时 watchdog、上电自检顺序、急停语义（卸力 vs 锁位，何时用哪个）。为什么安全层必须不依赖策略网络？
　🎯 考点：功能安全、独立保护层、失效安全设计。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **分层**：驱动器固件层（电流/温度/位置硬限）→ 主控安全层（本题）→ 策略层。每层假设上层会出任意错误。
- **软限位**：关节位置接近机械限位按裕度衰减力矩、速度/力矩超限直接钳制——在动作下发路径上无条件过滤，不管动作来自谁。
- **姿态超限卸力**：IMU 判倾角 > 阈值（如 45°）→ 进入安全态。**急停语义**：摔倒中**卸力**（damping 模式，电机只阻尼不出力——锁位会让倒地的机器人和地面硬抗，烧电机断腿）；静止站立故障 → **锁位/缓降**更安全。即"动则卸力、静则保持"。
- **通信 watchdog**：策略进程心跳超时（如 3 拍 60ms 无新动作）→ 电机切默认阻尼态；电机反馈超时 → 同样降级。
- **上电自检**：总线枚举 12 电机 → 逐电机读位置且在合理范围 → 当前姿态接近趴姿才允许站立流程（本仓库 robot_web 的安全校验同思想）→ 小电流测试每关节响应方向。
- **不依赖策略网络的原因**：策略是**被监护对象**——NaN、分布外输出、进程崩溃都是它的合法故障模式，让被监护者自我监护在逻辑上不成立；安全层必须是简单到**可被穷举验证**的确定性代码（无浮点网络、无动态内存），且运行在独立的执行路径（理想是独立 MCU）上。

</div>
</details>

**Q84 🟡** onnxruntime 在嵌入式 ARM 上推理 `[256,256,128]` MLP：图优化、线程数、内存 arena、fp16 各能带来什么？怎么 benchmark 真实控制环里的 P99 推理延迟而非平均值？
　🎯 考点：推理优化、尾延迟。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- 这个网络极小（~12 万参数，单次推理 ~0.25 MFLOPs），瓶颈不在算力在**调度开销与抖动**：
- **图优化**（ORT_ENABLE_ALL）：算子融合（Gemm+ELU）、常量折叠——小网络下融合省的是每算子调度开销，收益比例可观。
- **线程数**：**设 1**！小矩阵多线程的同步开销远大于并行收益，且多线程引入调度抖动——单线程绑核（配合 Q81 的隔离核）延迟最低最稳。
- **内存 arena**：预分配避免推理路径上 malloc——杀掉偶发的分配慢路径（直接削 P99 尾巴）。
- **fp16**：ARM NEON fp16 吞吐翻倍，但这个尺寸本来就 <100μs，意义不大；且需重新过 Q43 数值一致性门（1e-4 的阈得放宽并重新验证）。
- **P99 benchmark**：不要孤立循环跑一万次取平均——要**在真实控制环里**（总线 IO、其他线程都在跑）每拍打时间戳，攒 ≥10⁵ 样本出直方图，报 P50/P99/P99.9/max；孤立 benchmark 的缓存是热的、核是空的，系统性低估尾延迟。

</details>

**Q85 🔴** 策略热更新：机器人运行中切换新 checkpoint，要处理哪些状态连续性问题（last_action 缓冲、观测归一化统计、循环网络 hidden、动作跳变限幅）？设计一个安全的在线 A/B 切换流程。
　🎯 考点：状态迁移、平滑切换、上线流程。

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

**Q86 🟡** "盲走"（纯本体感知）与"感知行走"（外感知高度图）的能力边界各在哪里？盲策略靠什么机制翻越 10cm 台阶（提示：接触反射、抬腿高度先验、快速恢复）？

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **盲走边界**：只能**反应**不能**预见**——能处理可踩实的随机粗糙地形、缓坡、低台阶（速度要慢、靠试探），不能处理需要选落脚点的稀疏支撑（梅花桩、缝隙）、需要提前调整的高障碍、视觉才能区分的陷阱（深色地毯 vs 坑）。
- **盲策略翻台阶的三个机制**：① **接触反射**——摆动腿撞上台阶沿，本体感知（关节力矩/速度突变）触发"绊倒反射"：立即加大抬腿高度重新摆动（训练中粗糙地形+DR 教出来的）；② **抬腿高度先验**——课程里见过台阶的策略默认摆动轨迹留高度余量，等于"宁可多抬不可踢到"；③ **快速恢复**——踩空/磕绊后的姿态恢复能力兜底，允许试错。本质是把"看见"换成"撞了再说+撞得起"。
- **感知行走**：高度图进 obs 后可**预见式**选落脚点、提前调步态——能上稀疏支撑与高障碍，代价是感知 sim2real gap（Q87）。

**Q87 🔴** 以机器人为中心的高度图（heightmap）作为 obs：采样布局、坐标系选取、训练时来自仿真真值而部署来自深度相机/激光建图——这条感知 sim2real gap 怎么随机化（噪声、空洞、漂移、延迟）？

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **设计**：以 base 为原点、**yaw 对齐**（水平旋转随机体、但不随 roll/pitch 倾斜——保持"重力对齐"才与高度语义一致）的网格采样，典型前密后疏（前进方向 1.2m×0.8m、0.05–0.1m 分辨率，~100–200 个点），每点值 = 地面高度 − base 高度。

**Q88 🟡** 地形课程（由 Isaac Lab terrain generator 的多 proportion 地形混合实现）如何与难度升降级联动？"踩级"机制（学得好的 env 升难度）相比固定混合比例的优势？

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **机制**：terrain generator 把地形按行排难度（行 0 平地 → 行 N 高台阶/深沟），列内同难度不同随机样式；每个 env 记录表现（episode 末走过的距离/跟踪误差），达标 → 下次 reset 出生到下一行，不达标 → 降一行。Isaac Lab 的 terrain curriculum 原生支持，本仓库 `--terrain curriculum` 即此。

**Q89 🔴** 高程图来自 SLAM/里程计时存在漂移与时延：策略对"图与脚不对齐 3cm"这种系统误差如何免疫？训练侧与估计侧各能做什么？

<details><summary>📖 参考答案</summary>

**Q90 🟡** 为本机器人训练"从任意倒地姿态翻身站起"策略：初始状态分布、终止/成功判据、与行走策略的衔接（状态机切换 or 统一策略）怎么设计？

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

**Q91 🔴** 训练原地起跳/跳越：力矩上限（Q41 的 12N·m 余量问题）如何制约跳跃高度？effort 模式是否比 position 模式更适合爆发动作？飞行相与触地相的奖励怎么分段设计？
　🎯 考点：爆发动作、执行器极限、相位化奖励。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **力矩→高度的物理账**：起跳高度 `h = v²/2g`，起跳速度由蹬伸段冲量定：`m·v = ∫F dt`。本机 11.7kg、`effort_limit=12N·m`、腿长 ~0.3m → 单腿峰值蹬力 ~40N×4 腿 =160N，减去重力 115N，净加速度仅 ~4 m/s²，0.2m 蹬程末速 ~1.3m/s → 理论跳高仅 ~8cm——**当前力矩上限基本锁死了跳跃能力**；放宽到 18N·m（todo 计划）净蹬力翻倍以上，跳高才进入 20–30cm 量级。爆发动作正是"effort_limit 裕量"最敏感的行为。
- **effort 模式的适配性**：跳跃蹬伸要的是"全程贴最大力"的 bang-bang 式输出——position+PD 模式下力矩是误差的副产品，要贴限只能命令很远的位置目标（行为怪异且增益敏感）；effort 直出力矩天然契合爆发段。折中方案：混合动作（支撑/蹬伸相 effort、摆动/空中相 position）或可变 PD 增益动作。
- **分相奖励**：蹬伸相——竖直速度/蹬地冲量奖励；飞行相——目标高度/姿态保持（别后仰）、收腿（过障）；触地相——**冲击惩罚**（峰值接触力/关节力矩）、缓冲屈膝奖励、落地后稳定恢复奖励。相位由接触状态机判定。不分相的单一"跳高奖励"几乎必然学出硬着陆。

<div style="background:#E8F1FD; border-left:4px solid #1E88E5; padding:10px 14px; border-radius:6px;">

**Q92 🟡** 单策略多步态（trot/pronk/bound 指令可切换）与多策略 + 状态机切换的取舍？步态指令在 obs 里怎么编码（one-hot vs 相位发生器参数）？
　🎯 考点：多模态行为、指令条件化。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **单策略多步态**：步态指令进 obs（同 Q17 的指令条件化），优势是步态间**过渡**由策略自己学（连续、自然）、部署只有一个网络；风险是模式干扰（学糊成"平均步态"）——需要奖励按指令切换（指令 trot 时用 trot 相位奖励）+ 各步态采样均衡。
- **多策略+状态机**：每个步态单独训到极致，互不干扰、可独立迭代；代价是切换瞬间的分布断裂（Q85 的热切换问题每次切步态都要过一遍）、组合爆炸（N 个步态 N² 个过渡）。**经验**：步态差异大（walk vs gallop）或安全关键时用多策略；常用连续谱（trot 变频变速）用单策略。
- **指令编码**：**one-hot** 简单但步态间是离散跳变，过渡全靠策略脑补；**相位发生器参数**（每腿相位偏移 φ_i + 频率 f + 占空比 d，配合 obs 里的 sin/cos 相位时钟）更优——trot/pace/bound 只是 φ 配置不同，步态空间连续可插值，策略学的是"跟随相位时钟"这一个技能，切步态 = 平滑改时钟参数，还免费获得频率/占空比在线可调。

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **纯 DR（鲁棒策略）**：训练时 base 质量随机 ±2kg+，策略学一个对所有质量都凑合的**保守**控制律——实现零成本，但"对所有人都好 = 对谁都不最优"：轻载时过于费力、重载时裕量吃紧（11.7kg 机器 +2kg 是 17%，对力矩本就零余量的本机尤其难，Q41）。
- **HIM 隐式自适应**：质量变化在本体感知历史里有清晰签名（同样动作下加速度响应变慢、支撑力矩变大），历史编码器能隐式辨识并**条件化**策略——每个质量下都接近各自最优，且无需额外传感器；代价是 Q70 的可调试性问题。
- **显式负载估计**：用动量观测器/支撑相力平衡在线估计总质量与 CoM 偏移，作为 obs 喂给（训练时以真值+噪声喂的）策略——可解释、可监控、还能给安全层用（超载告警）；代价是估计器本身的工程量与其误差进入闭环。
- **推荐组合**：DR 打底（保证估计错时不摔）+ HIM/显式估计提供自适应（恢复性能）；显式估计值同时供监控与安全层。即"鲁棒性靠分布训练、最优性靠在线辨识"。

**Q94 🔴【系统设计】** 给一台全新四足（只有 URDF 和电机数据手册）从零搭建 RL 运控栈，列出第一周的工作排序与每步的验收标准（资产转换/物理参数核对/单 env 可视化/奖励最小集/并行化/首条策略/sim2sim）。

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

（每步含验收门，过不了不进下一步）

| 天 | 工作 | 验收标准 |
|---|---|---|
| D1 | **资产转换与物理参数核对**：URDF→USD/MJCF；逐 link 核 mass/inertia/CoM（对数据手册/CAD），核关节限位/轴向/零位 | 总质量与实物 ±2%；每 link 惯量正定合理；无嵌入 PhysicsScene（Q35） |
| D1–2 | **单 env 可视化 + 物理 sanity**：默认姿态自由落体、重力下站姿 PD 保持 | 落体不爆炸不穿地；PD 撑住站姿不抖；力矩量级对得上静力学手算（dynamics.py 同款递推） |

| D3–4 | **并行化与吞吐**：scale 到 2–4k env，GPU caps 按显存调（Q36） | 吞吐达标（>5 万 steps/s 量级）；72h 无 invalid_state 风暴 |
| D4–5 | **首条策略**：平地 PPO，标准超参（本仓库那套）不调参先跑通 | 能走、不摔、肉眼步态不病理；分项曲线无主导异常 |

**原则**：每天结束有可运行 artifact；物理正确性问题（D1–2）绝不带进训练阶段——RL 会把建模错误学成"特性"，后期翻倍还债。

**Q95 🔴【估算】** 估算本机器人 trot @1m/s 的运输成本 CoT = P/(mgv)：从关节力矩×角速度积分得机械功率，加上电机铜损，量级大概多少？与动物（~0.3）和优秀四足（~0.5）比差在哪？
　🎯 考点：能效估算、功率分解。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **定义**：`CoT = P/(mgv)`，本机 mgv = 11.7×9.81×1.0 ≈ **115W**——即 CoT 每 1.0 对应 115W 总功耗。
- **机械功率**：eval 实测（model_750）平均机械功率 **~515W** → 机械 CoT ≈ 4.5（已是仓库实测值）。这个数极高，根因是 Q41 的"贴限硬走"+蹲姿（base 0.256 vs 目标 0.32，深蹲姿态静态力矩翻倍）。
- **铜损补充**：`P_cu = Σ(τ_rotor/Kt)²R`。shank 类关节 rms ~8N·m → 转子侧 ~1.3N·m，按 GO-M8010-6 量级 Kt~0.07N·m/A（转子侧）、R~0.1Ω 估：单电机大力矩关节 ~30W 上下，12 电机合计可再添 **100–200W** → 总 CoT 可能逼近 **5–6**。
- **对比与差距来源**：狗 ~0.3、优秀四足（ANYmal/Go 系列）0.4–0.8。差 10 倍的去向：① 贴限高力矩行走（τ² 损耗对力矩平方敏感）；② 蹲姿支撑（几何劣势）；③ 无力矩惩罚 → 策略毫无省力动机（torque penalty 重训正是 todo 首项）；④ 动物有弹性储能（肌腱），刚性机器人每步动能耗散。**预期**：抬高站姿 + 力矩惩罚 + 指令条件化慢速步态，CoT 降到 1–2 是现实目标。

</div>
</details>

**Q96 🟡** 设计一组"策略质量"评估指标，超越平均奖励：速度跟踪误差分布、摔倒率/MTBF、足端打滑量、力矩 RMS 与饱和占比、步态相位一致性、抗推搡恢复成功率。为什么上线决策不能只看 reward？
　🎯 考点：评估体系、reward 与真实目标的错位。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **指标集**（本仓库 `eval.py` 已实现大部分）：

| 维度 | 指标 | 抓什么问题 |
|---|---|---|
| 可靠性 | fall_rate、mean_survival_s、抗推搡恢复成功率 | 上线安全底线 |
| 任务 | vx_rmse / yaw_mae 的**分布**（不止均值） | 跟踪质量与长尾 |
| 执行器 | per-joint τ rms / `near_limit_frac` | 真机可兑现性（Q41） |
| 能效 | mean_power_w、CoT | 续航与发热（Q95） |
| 步态 | duty_factor、touchdown_hz、diag_inphase_rate、foot_slip | 病理检测（bound/滑步） |
| 鲁棒 | 摩擦/质量/噪声/延迟扫描下以上指标的衰减曲线 | 泛化边界 |

- **reward 不够的原因**：① reward 是**训练的手段**不是工程目标——它是十几个项的加权和，4.5 的 CoT 和 84% 力矩饱和在 reward 里可以完全不可见（本仓库实锤：reward 漂亮、速度跟踪近完美，但靠贴限硬走换来，Q41）；② reward hacking 的定义就是"reward 高但行为错"——用 reward 验收等于让被考核者出考题；③ 上线决策要的是**最坏情况**（P99 跌倒率），reward 是均值型量。

**Q97 🔴【场景】** 真机连续运行 10 分钟后步态逐渐变差最终趴下，重启后恢复。列出按时间常数排序的嫌疑（电机温升降额、驱动器过热限流、电池电压下降、结构松动、里程计漂移累积），以及每个的判别实验。
　🎯 考点：慢时变故障、按时间尺度排查。

</div>

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

（"重启恢复"指向**状态积累类**故障；按时间常数从分钟级排起）

1. **电机/驱动器温升降额**（热时间常数正是分钟级，头号嫌疑）：判别——读电机/驱动器温度遥测画时间曲线，对比力矩指令 vs 实际电流响应比随时间衰减；趴下时摸最热关节（shank 类，Q41 的 84% 饱和关节发热 ∝τ²rms 最猛）。**重启恢复=散了几分钟热**，完全吻合。
2. **电池电压下降**（放电曲线 + 大电流内阻压降）：判别——母线电压全程录波，看趴下前是否进入低压区/大负载掉压加深（Q76 包络收缩）；换满电电池看故障时间是否推迟。注意重启恢复对它**部分**吻合（休息后电压回弹）。
3. **驱动器过热限流**（保护阈值触发，比电机热更突然）：判别——读驱动器故障寄存器/限流标志；症状是"渐变后突然断崖"。
4. **结构松动**（螺丝/连杆间隙渐进，时间常数更长且**重启不恢复**——用于排除）：判别——运行前后手动摇各关节量间隙；若重启立即恢复基本排除。
5. **里程计/估计漂移累积**（若部署用了显式估计）：判别——静止时看估计速度是否非零漂移；但纯本体盲策略（本仓库现状）无此环节，排序靠后。
- **方法论**：按"重启是否恢复 + 恢复需要多久"切分——秒级恢复 = 软件状态，分钟级 = 热，换电池才恢复 = 电源，永不恢复 = 机械。一次只动一个变量。

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **直接复用**：训练管线全套（PPO/rsl_rl/并行 env/eval 体系/DR 框架/导出链/数值防护 Q31Q38）、obs 设计模式（proj_gravity/残差动作/last_action）、方法论（课程/分项诊断/sim2sim 验证）。

**Q99 🟡** 团队协作：奖励权重改动如何做到可追溯、可回滚、可对比（实验命名规范、配置即代码、基线锁定、每改动一条曲线）？为什么"在同一个 outputs 目录里反复覆盖"是事故之源？
　🎯 考点：实验管理、变更追溯。

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

- **配置即代码**：权重只活在版本控制的 cfg 文件里，禁止跑前手改不提交——每次训练自动 dump 完整 cfg + git commit hash + 未提交 diff 进 run 目录（rsl_rl 默认行为，保住它）。
- **命名与台账**：run_name 编码意图（`torque_2e-4_vs_baseline`）；每次实验在台账立项（本仓库 `EXP-XXXX` + `tools/exp/log_run.py`：假设/改动/结论三段式）——回答"当时为什么这么改"。
- **基线锁定**：当前最优 checkpoint + 其 cfg 打 git tag，任何改动与锁定基线对比（同 seed 组、同 eval 套件，Q65）；优于基线才晋升为新基线。**一次只改一个权重**（Q32），曲线对比在 swanlab 同 project 下分组。
- **覆盖目录的事故链**：① 旧结果被静默覆盖 → "上周还好的"无法复现、没法 diff 是代码还是权重变了；② checkpoint 与产生它的 cfg 失联 → 部署了不知道是什么配置训的策略（真机事故直接来源）；③ 多人并发互相踩文件。时间戳目录 + 只追加不覆盖（本仓库 `logs/rsl_rl/<exp>/<时间戳>_<run_name>` 结构）从机制上消灭这一类事故。

**Q100 🔴【开放】** 展望：端到端 RL、MPC+WBC、以及大模型/世界模型驱动的运控（如视频预训练、跨形态基础模型）各自的天花板在哪？你认为 3 年内四足运控的工程主流形态是什么？给出论据而非口号。
　🎯 考点：技术判断力、趋势论证。

<details><summary>📖 参考答案</summary>
<div style="background:#EAF6EC; border-left:4px solid #43A047; padding:10px 14px; border-radius:6px;">

（示例论证，面试看的是论据结构而非站队）

- **端到端 RL 的天花板**：可验证性——无法对分布外行为给出保证，安全关键场景（载人、高价值硬件）始终需要包一层可证安全壳；以及每换形态/传感器重训的边际成本。但在"训练分布能覆盖的任务"内，其鲁棒性与工程简洁性已碾压（四足行走基本被 RL 终结，工业产品 Go2/ANYmal 实际出货的 locomotion 都是 RL）。
- **MPC+WBC 的天花板**：模型保真度与在线算力——接触丰富、模型时变的场景（软地、负载变化）失配即失效；建模与调参的人力成本不随出货量摊薄。它的不可替代位置在**可证约束**（力矩/摩擦锥/避碰硬保证）。
- **大模型/世界模型的天花板**：50Hz 力矩级控制的实时性与数据效率——视频预训练给的是语义/任务先验，不是接触动力学的毫秒级反馈；3 年内更可能落位在**上层**（任务规划、技能选择、地形语义）而非替代底层控制律。
- **3 年主流判断**：**分层混合**——底层 RL 步态策略（吃鲁棒、便宜）+ 安全层硬约束（吃保证，Q83 的进化版：从规则壳到 CBF/可达集）+ 上层学习式规划（落脚点/技能切换，逐步吃进大模型能力）。论据：① 这是当前头部产品（ANYmal 感知行走、Atlas 电动版路线公开信息）已收敛的形态，工程惯性大；② 各组件失效模式互补（Q58），没有单一范式能同时给出鲁棒+保证+泛化；③ 算力趋势（机载 GPU 普及）正好够"RL 推理 + 轻量在线 QP"而不够全身大模型 50Hz 闭环。
