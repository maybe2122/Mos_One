

## 1️⃣ 如何判断四足机器人打滑

### (A) 足端速度 vs 机器人速度

* 理论上，着地的足端速度应当相对于地面接近 **零**（no-slip condition）。
* 可以计算每条腿的足端速度：
  [
  v_\text{foot} = J_\text{leg} \cdot \dot{q}
  ]

  * `J_leg` 是腿端雅可比矩阵
  * `q̇` 是关节速度
* 如果 `|v_foot|` 很大 → 足端在滑动（打滑）

### (B) 机器人运动不一致

* 机器人期望前进速度 vs 实际速度差距很大
* 踏步周期中，腿伸出地面却机器人位移没有增加 → 打滑

### (C) 地面接触力监测

* 如果你的模拟器 / 传感器可以输出接触力：

  * 监测 `f_z`（垂直力）和摩擦力
  * 足端摩擦力达到极限但仍移动 → 打滑

---

## 2️⃣ 影响四足机器人打滑的因素

| 因素               | 说明                                 |
| ---------------- | ---------------------------------- |
| **摩擦系数 μ**       | 地面摩擦力太小 → 足端容易滑动                   |
| **接触力不足**        | 足端垂直力小 → 摩擦力 F = μ * Fz 小          |
| **步态 / 踏步速度过快**  | 快速步伐或大加速度 → 足端无法保持静止               |
| **控制器设计**        | torque/position 控制不合理 → 过大腿端速度导致滑动 |
| **机器人重量 / 质量分布** | 重心不稳 → 足端压力分布不均匀                   |
| **足端接触模型**       | 模拟器或现实中足底面积太小 → 摩擦力不足              |

---

## 3️⃣ 如何改正 / 减少打滑

### (A) 物理参数调整

* **增加摩擦系数** μ
* **增加接触力**：调节步态使足端承受更多垂直力
* **增加足底接触面积**（模拟或机械设计上）

### (B) 控制层面

* **降低步频 / 减小步幅** → 足端有足够时间与地面摩擦
* **使用 torque control / admittance control** → 避免关节快速运动导致足端滑动
* **步态优化**：Crawl 或 Trot 步态比 Bounding 或 Gallop 更稳
* **加入摩擦约束或正则化**：在 RL reward 中 penalize 足端滑动速度

### (C) RL训练层面

* **奖励 shaping**：

  * 奖励足端静止性（相对于地面速度接近 0）
  * 奖励足端接触力合理分布
* **模拟器物理调整**：

  * 避免 unrealistically low friction
  * 调整 solver iterations / constraint iterations 提高足端稳定性

---

💡 **总结**：

* 四足机器人“打滑”本质上是**足端摩擦力不足导致机器人实际速度低于理论速度**。

* 判断方法：

  1. 足端速度 vs 地面 → 不为零 → 打滑
  2. 机器人速度低于期望速度
  3. 接触力达到摩擦极限但仍移动

* 改进方法：

  * **物理**：增加摩擦、足底面积
  * **控制**：降低步幅/步频、使用 torque control
  * **训练**：奖励足端静止性、优化步态

“一跳一跳”的四足步态，其实不一定只是**对称性奖励没做好**，更多时候是：系统在用“弹跳模式”来最大化速度/效率，因为它觉得“跳”比“走”更容易拿分。

对称性只是其中一小块，更关键的是下面这些“让它别乱跳”的约束。

---

# 🧠 先说结论：跳跃步态的真正原因

通常是三类问题叠加：

### ① 动态过激（最常见）

* action scale 太大
* PD 太硬 / damping 不够
* torque 惩罚太弱

👉 结果：机器人“蹦着走更省事”

---

### ② 没有“步态节律”

* 没有 gait phase
* 没有 foot contact pattern bias

👉 结果：它自己发明“跳跃最优解”

---

### ③ 脚落地约束太弱

* foot slip penalty 不够
* contact timing 没约束
* 没有 stance/swing 区分

👉 结果：它选择“全脚一起弹”

---

# ✅ 除了对称性，还应该加这些（重点）

下面这些比“对称 reward”更有效：

---

# 1️⃣ Foot Air Time Reward（非常关键）

鼓励“合理腾空时间”，避免全脚一起跳

```text
reward += min(air_time, target_air_time)
```

效果：

* 抑制“同时起跳”
* 形成 trot / walk 节律
* 步态会自然分化

👉 这是防“兔子跳”的核心

---

# 2️⃣ Contact Timing Reward（强烈推荐）

定义 gait phase：

* 哪些脚该在地上
* 哪些脚该在空中

例如 trot：

```text
LF + RR 同步
RF + LR 同步
```

奖励：

```text
if expected_contact == actual_contact:
    +reward
```

👉 这是“步态正确性”的核心约束

---

# 3️⃣ Foot Slip Penalty（必须有）

```python
penalty = foot_vel_xy_when_contact
```

跳跃机器人经常：

* 落地滑一下再弹
* 或者“蹭地跳”

这个 reward 不够就会出现“弹球式 gait”

---

# 4️⃣ Base Height Stability（抑制跳跃本体）

非常关键！

```python
(base_height - target_height)^2
```

或者：

```text
Δz_base penalty
```

👉 直接打击“上下弹跳”

---

# 5️⃣ Vertical Velocity Penalty（核心）

这是很多人忽略的关键：

```python
reward -= base_lin_vel_z^2
```

效果：

* 抑制起跳
* 抑制落地冲击
* 强制“水平走路”

---

# 6️⃣ Action Rate Penalty（防弹簧行为）

```python
(a_t - a_{t-1})^2
```

没有这个：

👉 policy 会学成“弹簧系统”

---

# 7️⃣ Torque / Energy Penalty（防爆发式跳）

```python
τ^2
```

或：

```text
power = τ * joint_vel
```

👉 防止“蹬一下跳很高更省 reward”

---

# 8️⃣ Gait Phase / Phase-based Observation（高级但非常有效）

加入：

```text
sin(phase)
cos(phase)
```

或者周期函数：

```text
phase = (t % T) / T
```

👉 作用：

* 强制形成周期性步态
* 防止随机跳跃策略
* 让“走路”成为最容易学的结构

---

# 9️⃣ Lateral Symmetry（你同事说的这个）

可以加，但要注意：

👉 它不是主要问题，只是“修饰项”

形式：

```text
LF ≈ RF
LR ≈ RR
left ≈ right
```

但：

⚠️ 如果没有 gait constraint，它仍然可能“对称地跳”

---

# 10️⃣ Stance / Swing Separation Reward（很有效）

定义：

* stance leg（支撑）
* swing leg（摆动）

奖励：

```text
stance leg velocity ≈ 0
swing leg clear trajectory
```

👉 这是“真正像狗走路”的关键

---

# 🚨 一个很重要的真相

如果你的机器人在“跳”：

👉 说明它认为：

> “跳 = 更快 / 更稳定 / reward 更高”

所以你需要做的是：

## ❌ 不要只“限制跳”

## ✅ 要“让走路更优”

# 🧪 推荐你优先改的 5 个（排序）

如果你只想快速修好：

### 🥇 1. vertical velocity penalty

### 🥇 2. base height stability

### 🥇 3. action smoothing

### 🥇 4. foot slip penalty

### 🥇 5. foot air time reward

这五个基本可以把“兔子跳”打掉 80%

# 🧭 一个实用判断标准

如果你看到：

* 身体上下起伏 > 前后移动
* 四脚同时离地
* 落地有冲击弹起

👉 说明：

> reward 在鼓励“飞行 gait”，而不是 walking gait
