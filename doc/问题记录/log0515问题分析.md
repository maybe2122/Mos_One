# 问题分析与修复记录

> 基于 `logs/rsl_rl/rsl_rl_mos2026_2_closed_usd_2026-05-15_08-02-25.json`
> 训练曲线（1000 个点，覆盖 step 0→1743，约 5.6 小时）的诊断报告。

---

## 1. 训练曲线现象

### 1.1 灾难性的数值尖峰

mean reward 曲线里出现了 **9 个量级远超合理范围**的负尖峰：

| step | value | 量级 |
|------|-------|------|
| 15 | −2.97e+00 | 小 |
| 177 | **−1.34e+04** | 万 |
| 199 | −3.03e+01 | 十 |
| 274 | **−1.46e+28** | 10²⁸（恐怖） |
| 295 | **−1.74e+08** | 亿 |
| 393 | −5.78e+01 | 十 |
| 596 | **−5.76e+04** | 万 |
| 1100 | −4.70e+01 | 十 |
| 1432 | −2.01e+00 | 小 |

正常 reward 在 **+10 ~ +20** 之间，这种 10²⁸ 量级的负值意味着 reward 计算里出现了未被拦截的极端数值。

### 1.2 学习曲线的实际形状

把尖峰剔掉，按 100-step bin 平均：

```
 0–99   : +3.0    起步
100–199 : +9.4
200–499 : +11~14  快速上升
500–1100: +12~13  ★ plateau 600 个 iter
1200–1700: +14~16 慢速恢复
```

- 学习曲线**本身能涨**，但 500–1100 段出现明显 plateau。
- plateau 的位置和前面几次大尖峰对应：PPO 被毒数据带偏 value/policy 之后要花 50~100 iter 才能洗回来。
- 即便忽略尖峰，episode_sum ≈ 15 也远低于理论上限 ≈ 360，说明策略本身没学会走路。

## 2. 根因分析

### 2.1 主因：闭链 closure joint 未启用 projection

USD 资产 `mos2026_2.usd` 里有 32 个 revolute joint，其中 **4 个**被标记为 `physics:excludeFromArticulation=True`：

| closure joint | 拓扑（body0 → body1） |
|---|---|
| `fl_close_loop` | fl_shank_link_b → fl_shank |
| `fr_close_loop` | fr_shank_link_b → fr_shank |
| `rl_close_loop` | rl_shank_link_b → rl_shank |
| `rr_close_loop` | rr_shank_link_b → rr_shank |

这 4 个 joint 关闭了每条腿的 4-bar 平行连杆机构。它们存在于 articulation 简化坐标树**之外**，PhysX 需要作为单独的约束求解。

**未启用 projection** 时（cfg 里 `projected_loop_joint_names=[]`），约束求解出错后锚点会逐步漂移，积累到一定程度突然反弹回正确位置，期间 root pose / joint state 会瞬间跳到极大的有限值。

### 2.2 次因：reward 函数没有给输入做净化

`mos2026_2_closed_usd_env.py:229` 的 `_get_rewards` 原本只有最后做了一次：

```python
reward = torch.nan_to_num(reward, nan=0.0, posinf=0.0, neginf=0.0)
```

`nan_to_num` 只能拦 NaN / ±Inf，**拦不住异常大的有限值**。当 PhysX 给出 `root_z = 1e+13` 这种数值时：

```
base_height_error = (1e13 - 0.32)² ≈ 1e+26
   * -10 (base_height 权重)
   * 0.02 (step_dt)
   ≈ -2e+25 per env
4096 个 env 平均之后 → -1e+28
```

正好对应 step 274 的 −1.46e+28。

### 2.3 雪上加霜：reward 权重失衡

原始 `reward_scales`：

```python
"base_height":      -10.0,    # 项无上界，权重又大
"lin_vel_z":        -0.5,     # z 速度上限 20 m/s，单步可贡献 -200
"track_lin_vel_xy": 4.0,      # tracking 信号被惩罚淹没
"track_ang_vel_z":  0.5,
```

- `lin_vel_z² ≤ 400`（PhysX `max_linear_velocity=20.0`），乘 −0.5 = −200/step，
  远大于 +4 max 的 tracking reward。前进信号被压制 → plateau。
- `base_height_error` 没有上界，是数值雷管。

### 2.4 配置层面的其它隐患

| 位置 | 配置 | 问题 |
|---|---|---|
| env_cfg.py:195,202 | `solver_position_iteration_count=64` | 迭代次数高，闭链求解不收敛时容易发散 |
| env_cfg.py:190 | `linear_damping=0.0`, `angular_damping=0.0` | 没有阻尼，约束断裂瞬间能量全保留 |
| env.py:223 | `obs = torch.nan_to_num(obs, nan=0, ...)` | obs 里某些项正常值不是 0，喂 0 进 policy 是 OOD 输入 |

## 3. 已做的修复

### 3.1 启用 closure joint projection

**`mos2026_2_closed_usd_env_cfg.py`**

```python
projected_loop_joint_names = [
    "fl_close_loop",
    "fr_close_loop",
    "rl_close_loop",
    "rr_close_loop",
]
auto_detect_loop_joints = True  # 未来换 USD 时自动列出 closure joints
```

env.py 里的 `_patch_projected_loop_joints` 会在这些 joint 上设置：

```python
physxJoint:enableProjection = True
physxJoint:projectionLinearTolerance = 0.002   # 2 mm
physxJoint:projectionAngularTolerance = 0.05   # ~3°
```

约束漂移超过阈值时 PhysX 会强制把锚点拉回，**从根上**阻止数值发散。

### 3.2 `_get_rewards` 加 invalid 屏蔽 + clamp

**`mos2026_2_closed_usd_env.py`**

```python
# 1) 先 detect invalid state（沿用 _get_dones 同款逻辑）
invalid_state = (
    ~torch.isfinite(root_pos_w).all(-1)
    | ~torch.isfinite(root_lin_vel_b).all(-1)
    | ~torch.isfinite(root_ang_vel_b).all(-1)
    | ~torch.isfinite(joint_pos_full).all(-1)
    | ~torch.isfinite(joint_vel_full).all(-1)
)

# 2) 原始信号 nan_to_num
root_pos_w     = torch.nan_to_num(root_pos_w, ...)
root_lin_vel_b = torch.nan_to_num(root_lin_vel_b, ...)
# ...

# 3) 平方项 clamp（防止异常大的有限值传播）
base_height_error = torch.clamp(torch.square(root_height - 0.32), 0, 4.0)
upright_error     = torch.clamp(..., 0, 2.0)
lin_vel_z         = torch.clamp(torch.square(...), 0, 100.0)
ang_vel_xy        = torch.clamp(..., 0, 100.0)
joint_vel         = torch.clamp(..., 0, 1.0e4)

# 4) invalid env 强制 reward = 0
reward = torch.where(invalid_state, torch.zeros_like(reward), reward)
reward = torch.nan_to_num(reward, ...)
```

### 3.3 Reward 权重重平衡

| 项 | 旧值 | 新值 | 理由 |
|---|---|---|---|
| `base_height` | −10.0 | **−3.0** | 配合 clamp(0,4) 后单步最坏 −12，不再主导 |
| `lin_vel_z` | −0.5 | **−0.05** | 与 +6 tracking 量级匹配 |
| `track_lin_vel_xy` | 4.0 | **6.0** | 强化前进信号 |
| 其他 | 不变 | | |

### 3.4 训练脚本新增 `--max_gpu_mem`

**`scripts/rsl_rl/train.py`**

新增参数（默认 32.0 GB），按比例缩放 PhysX 的 5 个 `gpu_*` 容量字段。小显卡示例：

```bash
python scripts/rsl_rl/train.py \
    --task StackForce-Mos20262ClosedUsd-ClosedUsd-v0 \
    --headless --num_envs 2048 --max_gpu_mem 16 \
    --max_iterations 1500 --run_name gpu16
```

## 4. 所有可调参数详解

### 4.1 Reward 相关 (`mos2026_2_closed_usd_env_cfg.py`)

| 参数 | 当前值 | 含义 | 数学上界 | 调参方向 |
|---|---|---|---|---|
| `reward_scales.alive` | 0.0 | 每步活着给的常数 | +1×scale | 设 0.5~1.0 让 policy 不轻易死，但容易陷入「站着不动」局部最优 |
| `reward_scales.upright` | 0.0 | `exp(-grav²/0.25)` 上身姿态 | +1×scale | 与 `flat_orientation` 二选一；exp 容易饱和 |
| `reward_scales.flat_orientation` | −2.5 | L2 姿态惩罚 `Σ grav²` | −2×scale | 调大 → 更挺直；过大压制探索 |
| `reward_scales.base_height` | −3.0 | 高度偏差平方 | clamp 后 −4×scale=−12 | 与 `base_height_target` 配合；调大让高度更稳 |
| `reward_scales.lin_vel_z` | −0.05 | z 速度² | clamp 后 −100×scale=−5 | 调大抑制跳跃 |
| `reward_scales.ang_vel_xy` | −0.02 | xy 角速度² | clamp 后 −100×scale=−2 | 调大抑制摇摆 |
| `reward_scales.joint_vel` | −0.0001 | Σ joint_vel² | clamp 后 −1e4×scale=−1 | 调大抑制颤抖，过大会让动作呆滞 |
| `reward_scales.action_rate` | −0.003 | Σ Δaction² 平滑性 | ~−0.1 量级 | 调大让动作连贯，过大会让 policy 不敢动 |
| `reward_scales.track_lin_vel_xy` | 6.0 | `exp(-err/σ)` 前进 tracking | +1×scale | 调大主导前进；过大忽视姿态 |
| `reward_scales.track_ang_vel_z` | 0.5 | `exp(-err/σ)` 转向 tracking | +1×scale | 调大让转向精确 |
| `tracking_sigma` | 1.0 | exp 的带宽 | — | 调小 → 严格、信号窄；调大 → 宽松、有梯度但收敛慢 |
| `base_height_target` | 0.32 | 目标身高 (m) | — | 与机器人腿长匹配 |
| `fall_height_threshold` | 0.22 | 低于此高度终止 | — | 调高让"趴下"立刻 reset |
| `fall_cos_threshold` | 0.85 | cos(tilt) 终止阈 | 0~1 | 调高更严格（向 1.0 接近 32°→0°） |
| `commanded_lin_vel_xy` | (1.0, 0.0) | 前进速度指令 (m/s) | — | 调小训练慢走，调大训练快走 |
| `commanded_ang_vel_z` | 0.0 | 转向速度指令 (rad/s) | — | 非零训练转向 |

## 5. 后续优化方向

按优先级排列：

### P0 — 立即可做

1. **跑一次重训**，看曲线是否还有尖峰、是否能突破 episode_sum ≈ 15 的 plateau。
2. **TensorBoard 看各 reward 子项**（`Episode_Reward/*`）确认哪个 term 主导。

### P1 — 训练稳定后

3. **降 solver 迭代数**：closure projection 启用后，`solver_position_iteration_count` 可以从 64 → 32，明显提速。
4. **加阻尼**：`linear_damping=0.01, angular_damping=0.01`，给约束断裂瞬间一点能量耗散。
5. **微调 reward 权重**：如果还是「趴着不走」，把 `track_lin_vel_xy` 再调到 8~10，`flat_orientation` 同步调到 −4 ~ −5。

### P2 — 性能进阶

6. **课程学习**：用 `--terrain curriculum` 让 policy 先在 flat 学会走，再渐进到 rough/stairs。
7. **域随机化**：在 `_reset_idx` 里给 mass / friction / 初始姿态加噪声，泛化更好。
8. **观测加噪 / delay**：模拟真机传感器，sim2real 必要。
9. **非对称 actor-critic**：critic 用 privileged obs（root state、接触力等），actor 只用本体感知 → 学习更快、转 sim2real。
10. **更大 `num_steps_per_env`**：当前默认 24 偏小，加到 48~64 可以让 value 估计更稳。
11. **更激进的 `tracking_sigma`**：开始 1.0 容易学，后期收紧到 0.25 强迫精确跟踪。

### P3 — 资产层面

12. **检查 closure joint 的 anchor 精度**：在 USD 里量一下 `*_close_loop` 的两端 anchor 是否在装配时严丝合缝；如果初始就有偏差，projection 也救不了。
13. **简化机械模型**：如果训练始终不稳，考虑把 4-bar 闭链用一个等效串联 joint 近似训（牺牲精度换稳定性），最后真机再换回闭链。

### P4 — 工程化

14. **加 wandb / mlflow 记录**：日志可对比。
15. **加 eval callback**：训练中固定 seed 跑确定性 episode，更可靠地衡量进展。
16. **多 seed 训练 + 求中位数**：避免单次 run 的运气因素。
